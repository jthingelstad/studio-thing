# workshop_bot — project memory

> Author-only Discord bot for *The Weekly Thing*. Helps Jamie assemble each week's issue and ships supporting bits (link curation, membership CTA, promotion drafts). One Python process, several `discord.py` clients (one per persona, own token/avatar), plus an APScheduler instance — all in the same asyncio loop.

Full design rationale: [`docs/workshop-content-loop-design-brief.md`](../../docs/workshop-content-loop-design-brief.md). The user-facing overview is [`README.md`](README.md). This file is the "what to keep in mind when working here" memory.

## Architecture: jobs are the spine

Every workshop_bot action — pulling content into the draft, reordering for the final, composing CTAs, drafting promotion copy — is a **job**: deterministic Python in `apps/workshop_bot/jobs/`, fired by the `/workshop …` slash surface (host: Eddy) and/or by cron. Some jobs make small encapsulated LLM calls (Eddy's draft review, the compose-* jobs, Marky's reports); most are pure Python. The job is the unit of scheduling and on-demand execution.

The slash surface is organised around the *content artifact*, not around the word "job": `/workshop issue {start,update,status,final,haiku,subject,cta,publish}`, `/workshop links scan`, `/workshop promo {prep,metrics}`, `/workshop campaign {add,edit,report,copy,sunset}`, `/workshop goal {set,done}`, `/workshop followup {list,add,cancel}`, `/workshop thingy {recent,show,sync}`, plus a top-level `/workshop status`. The internal job names used below (and by the scheduler, the `job_locks` table, and `agent_runs`) are the dash-cased ones — `update-draft`, `create-final`, `compose-haiku`/`-meta`/`-cta`, `build-publish`, `pinboard-scan`, `promotion-prep`, `daily-metrics`, `thingy-watch`, `follow-up-sweep`, etc.; the slash names are the operator-facing aliases (`/workshop issue update` → `update-draft`, `/workshop issue subject` → `compose-meta`, `/workshop links scan` → `pinboard-scan`, `/workshop promo metrics` → `daily-metrics`, `/workshop goal done` → `goal-achieved`, `/workshop thingy sync` → `thingy-watch`, and so on).

`jobs/_base.py` is the runtime: `JobContext` (carries `deps` — corpus/registry/team — and a `trigger` label; has `channel()` / `post()` helpers), `JobResult` (`{ok, message, data}` — `message` is acked to the invoker), single-asset locking (`job_lock([assets], name)` context manager backed by the `job_locks` SQLite table — two jobs that write the same file can't overlap; a lock held by a dead pid is stolen since a restart gets a new pid), and the draft-block helpers (`replace_block` / `get_block` for the `<!-- block:NAME -->` … `<!-- /block:NAME -->` markers in `draft.md`).

### The jobs

| Job | Trigger | What it does |
|---|---|---|
| `start-issue <n> <pub-date> <days>` | manual | Records the issue window in `workshop.db`, refreshes the `weekly-thing/workshop.json` pointer Shortcuts read (issue number, dates, predictable workspace URLs), seeds `draft.md` from `templates/draft_starter.md`, auto-fires `update-draft`. The only job that takes the issue number explicitly. |
| `update-draft` | daily 17:00 CT + after `start-issue` + manual | Pure projection: rebuilds `draft.md` from `templates/draft_starter.md` every run (so the layout always tracks the template) and fills each block — Notable/Briefly from Pinboard, Journal from micro.blog, `intro.md` / `haiku.md` verbatim from the authored asset files, the cover caption/date/location from `cover.json` (`{"caption":…,"location":…,"timestamp":…,"alt":…}` rendered as `caption\n\ntimestamp  \nlocation`, native `<img alt="…" src=".../cover.jpg" />` prepended — preferred) or a legacy verbatim `cover.md` (`_cover.render` / `_cover.alt`), and `Currently` from `currently.json` (a `{"Label":"value",…}` object rendered as `**Label:** value` lines — preferred) or a legacy verbatim `currently.md` (`_currently.render`). Output is shaped like a delivered issue (see "issue-markdown shape" below): `---`-fenced blocks, the Notable "discuss on Reddit" line, `### [Title](url)` link headings, `→ **[Title](url)**` Briefly form, elevated (titled) Journal posts, the `A haiku to leave you with…` close. Idempotent; replaces block content wholesale, nothing on the old draft preserved. **Alt-text auto-fill:** any rehosted journal image *and* the cover image whose alt is missing/empty gets one filled via a Claude vision call (`tools/alt_text.py`, sonnet, ≤120 chars, factual, no "photo of" preamble); cached forever in `image_alt_cache` keyed by content-addressed filename (`428e3db12e.jpg`) / `cover-{N}`, so the daily re-run is free once each image is seen. `cover.json.alt`, if set by the operator, wins over the vision call. Per-run cap of 15 calls (`WORKSHOP_ALT_VISION_CAP`); cap-exhaustion or generator failure leaves alt empty — Eddy's Hygiene review surfaces those. Then: a **solid editorial review** of the draft (`prompts/eddy/draft-review.md`, sonnet, every run) gets embedded in `draft.html` behind a "Show review" toggle — hidden by default, so the shareable link reads clean until someone flips it on (`_draft_review` → `render.render_and_upload_html(review_md=…)`); writes a `draft_digests` row; on Tue–Fri also runs Eddy's `#editorial` post-update card (`prompts/eddy/update-review.md`), silent Sat/Sun/Mon. Refuses (loudly) if `final.md` exists — the issue is locked. |
| `issue-status` | manual | Read-only section + asset completeness report. |
| `create-final` | manual | Eddy proposes a reordered/curated final (fenced markdown block); Jamie ✅/❌/🔄; writes `final.md`; posts "next: run compose-haiku/meta/cta, then build-publish". No auto-chain — each downstream job is run on demand and refuses-with-a-list until its prerequisites exist. |
| `compose-haiku` | manual | Eddy returns JSON haiku options → `#editorial`; Jamie picks → `haiku.md`. Required for ship. |
| `compose-meta` | manual | Two LLM steps in `#editorial`. **Subject:** Eddy returns **5 `WT<N> — <theme>` options** (`prompts/eddy/compose-subject.md` — Jamie's prompt, verbatim) → Jamie reacts 1️⃣–5️⃣ (🔄 = refresh). **Description:** Eddy generates a single comma-separated topic line (`prompts/eddy/compose-description.md` — also Jamie's prompt, verbatim) — *no picker*, the prompt is deterministic enough (length window, ordering rules, style guardrails) for one-shot. The success post surfaces both so Jamie can spot a bad description and edit in Buttondown or re-run. Writes `metadata.json` (subject/description generated; image/slug/number/publish_date deterministic). Required for ship. |
| `compose-cta` | manual | Patty decides 0/1/2 CTAs, 1–2 framings each, in **Thingy's** voice; → `#supporters`; Jamie picks per slot → `cta-1.md` / `cta-2.md` (each with `placement:` YAML frontmatter). Optional. |
| `build-publish` | manual | Assembles `publish.md` — byte-shaped like a real Buttondown body (the thing `pipeline/content/content.py publish` pushes). Order: a leading `<!-- buttondown-editor-mode: plaintext -->` glommed onto the intro → `## Currently` (if present) → the cover block (image + `cover.json`/`cover.md` caption via `_cover.render`) → each non-empty `## Header\n\n{content}` section (Notable/Journal/Briefly; empties drop out) → CTAs, each wrapped in the membership-block Liquid scaffold (premium / regular + the two Stripe Payment Link buttons / else + `{{ subscribe_form }}`) → `A haiku to leave you with…` + the bold/hard-break haiku + the "discuss on Reddit" closing line + the email-only `{% if medium == 'email' %}…tinylytics pixel…{% endif %}`. Parts are `---`-fenced; no `<!-- block:… -->` markers. Also writes `publish.html` — a Liquid-stripped regular-subscriber rendering (best-effort preview). Refuses with a missing-list (→ `#editorial`) if any required asset is absent: `final.md`, `haiku.md`, `metadata.json`, `intro.md`, `cover.jpg`. |
| `pinboard-scan` | every hour 07:00–22:00 CT, year-round; manual | Linky's per-link research pass. One **toread** lane (Jamie's own public-`toread` bookmarks not yet in `pinboard_research_done`, via `pinboard.toread_public_unresearched`) plus N **discovery feeds** declared in the `DISCOVERY_FEEDS` registry in `jobs/pinboard_scan.py` — each entry is a `FeedSpec` from `tools/feed_registry.py` carrying name, label, pin-label, fetch callable, per-scan cap, feed limit, and primary priority. Today's discovery set is IndieWeb News, Tildes ~tech, Lobste.rs, Hacker News (front page via Algolia), Pinboard popular. Two cross-source signals run on top of the dedup table: an **in-scan merge** (same URL on multiple feeds in this hour → one card, all discussion-thread links in the header, "Also trending on …" in the LLM's user message; primary picked by `primary_priority`) and a **cross-day uplift** (URL was seen on a *different* feed days ago → `## Cross-source uplift` block carries the prior sightings + original verdict; Linky reconsiders, capped at `_UPLIFT_PER_SCAN_CAP=5` per scan; sightings tracked in the new `popular_seen_sightings` table). To add a feed: write `tools/<name>.py` with `top(limit=N)` (or equivalent) returning `[{url, title, [discussion_url], [score], [comment_count], [submitter], [tags]}]`, add one `FeedSpec` line, append the name to `db.RESEARCH_SOURCES`. The job (not the LLM) iterates candidates and calls Linky's `core` once per link with `prompts/linky/research-card.md` (source-agnostic — switches on `toread` vs. discovery via the `## The link` block's `Source` field); each per-link call returns either a Discord card or one of two signals — `SKIP: <reason>` (discovery sources only — judged not interesting; URL is marked seen and never resurfaces from the same feed, may still uplift if a *different* feed surfaces it later) or `FETCH_FAILED: <reason>` (any source; URL is NOT marked, so it can retry next scan). Each card posts as its own `#research` message; the `discord_message_id` is recorded in `linky_research_messages` so a Discord reply / save-reaction can route back to the right Pinboard bookmark. URL normalisation for cross-source dedup is `tools/url_normalize.py`'s `dedup_key` (lowercases host, strips trailing slash, drops `utm_*`/`fbclid`/etc. — in-memory comparison only; stored URLs are unchanged). PASSes silently when all sources come back empty (common off-hour). |
| `promotion-prep` | auto on RSS detection; manual | Marky drafts syndication content (LinkedIn + r/WeeklyThing megathread + per-link threads, 2–3 framings each) for the latest *published* issue's `publish.md` → `#promotion`. Never auto-posts. |
| `daily-metrics` | daily 19:00 CT; manual | Polls active campaigns (Tinylytics `?ref=` traffic + Buttondown ref signups → a `campaign_metrics` row each run), checks subscriber growth + engagement; PASSes silently when nothing material moved, else Marky composes a terse report → `#promotion`. |
| `add-campaign <name> <ref> [signups] [traffic] [copy]` | manual | Inserts a row into the `campaigns` table. `ref` is stored verbatim (case-sensitive — Tinylytics matches the `?ref=` value exactly). `copy` (optional) is the actual promo text that ran in the placement. |
| `campaign-edit <name> [ref] [started_at] [ends_at] [signups] [traffic] [notes] [copy]` | manual | Updates an existing campaign's editable fields in place — only the ones you pass (`db.CAMPAIGN_EDITABLE`); the name is immutable and `status` flips via `campaign-sunset`. In `jobs/ops.py` → `db.update_campaign`. |
| `campaign-report` | manual | Active campaigns + current performance vs expected, each with its recorded copy. |
| `campaign-copy <name> [text]` | manual | Sets (or, with empty text, clears) a campaign's `copy` — the placement creative, so perf can be read against it. In `jobs/ops.py`. |
| `campaign-sunset <name>` | manual | Sets a campaign's status to `sunset` — `daily-metrics` stops polling it. |
| `set-goal <kind> <value> [notes]` | manual | Opens a new active goal in the `goals` table (`kind` ∈ `members`/`dollars`). Refuses if one's already active (close it with `goal-achieved` first — the table allows one row with `achieved_at IS NULL`). |
| `goal-achieved [notes]` | manual | Marks the active goal achieved (today); `notes` is appended to whatever was recorded at `set-goal` time. |
| `thingy-watch` | hourly (cron `7 * * * *`); manual via `/workshop thingy sync` | Pulls newly-logged Thingy conversations from the Lambda (`thingy_client.fetch_conversations` → the auth endpoint's `list_conversations` action), groups turns into conversations (same `subscriber_hash`, turns within ~30 min / a fresh browser history), has **Eddy** write a two-sided assessment (`prompts/eddy/thingy-review.md`, one-shot sonnet, JSON `{topic,reader,thingy,takeaway}`), mirrors each *new* one into `thingy_conversations` (stable local id, outlives the Lambda's ~60-day DynamoDB TTL), and posts a card to `#chatter`. Dedupes on turn `request_id`; PASSes silently when nothing new; drains a backlog ≤25 convos/run, ≤6 cards/run (the rest land next run, with a "+N more" line). `jobs/thingy.py`. |
| `thingy recent [count]` / `thingy show <id>` | manual | Read-only DB reads over `thingy_conversations` — `recent` lists the last N (one line each); `show` posts the assessment card and attaches the full transcript as a `.md` file. Also in `jobs/thingy.py`. |
| `follow-up-sweep` | hourly (cron `23 * * * *`) | Fires due **follow-ups** — commitments an agent (or Jamie) made via `followup__schedule` / `/workshop followup add`, stored in the `follow_ups` table. Two trigger kinds: time-based (`due_at` — any distance: tomorrow, next month, next year) and issue-based (`trigger_issue` — fires once the active in-flight issue reaches it). For each due one: runs the persona's agent loop with the note + a `## Today` context block, posts the agent's check-in to the persona's home channel, marks it fired. PASSes silently when nothing's due; an agent PASS marks it fired without posting; an agent/channel error leaves it open to retry; ≤8/sweep. `jobs/follow_up.py`. **This is the deliberate, targeted replacement for per-persona heartbeats** — it only does anything when a real commitment comes due. `jobs/follow_up.py` also backs `/workshop followup {list,add,cancel}` and the `followup__schedule` / `followup__list` / `followup__cancel` agent tools. |

Plus one top-level subcommand outside the artifact groups: **`/workshop status`** — a read-only DB-only ops snapshot (active issue window, active goal/campaigns, any held `job_locks`, the last few `agent_runs`). Distinct from `/workshop issue status` (the in-flight issue's *content* completeness) and `/workshop campaign report` (campaign perf vs expected). Source: `jobs/status.py`; `set-goal`/`goal-achieved`/`campaign-sunset`/`campaign-copy`/`campaign-edit` are in `jobs/ops.py` (tiny, no-LLM).

Scheduler: `scheduler/jobs.py` declares the cron `JobSpec`s; `scheduler/handlers.py` has `content_job` (the bridge cron→jobs, wired as `functools.partial(content_job, job="<name>")`) and `rss_check` (poll the feed, dedupe via `agent_notes`, auto-fire `promotion-prep`). There are **no per-persona heartbeats** — everything an agent does on a cadence is a job. The closest thing to "an agent acting on its own" is `follow-up-sweep` firing a commitment the agent itself scheduled.

Slash dispatch (`personas/commands.py`): the fast jobs `defer` → run → ack with the result (the followup send is wrapped in try/except — a Discord interaction token only lasts ~15 min). The **interactive** jobs (`create-final`, `compose-haiku`/`-meta`/`-cta` — they post options to a channel and wait for Jamie's reaction, which can outlast the token) instead ack *immediately* ("started — react in #editorial / #supporters") and then run; the job posts its own outcome to the channel, so no second followup is needed. `/workshop status` is attached directly to the `workshop` group rather than to one of the artifact subgroups (it's a bot-health view, not tied to a single artifact) — Discord allows a command group to hold both a subcommand and subcommand-groups.

### The unified asset pattern

Every piece of issue content is a standalone file in the S3 workspace `s3://files.thingelstad.com/weekly-thing/{N}/`:

- `intro.md` (the opener prose) — Jamie writes (Drafts → Shortcut); `update-draft` reads it verbatim into its block. The cover block's caption/date/location comes from `cover.json` — `{"caption":…,"location":…,"timestamp":…}` rendered as `caption\n\ntimestamp  \nlocation` with the derived `![](.../cover.jpg)` prepended (`jobs/_cover.py`); a legacy verbatim `cover.md` (caption + `date  \nlocation`) is still accepted. The optional "Currently" section comes from `currently.json` — a `{"Label":"value",…}` object (e.g. `{"Listening":"…","Watching":"…"}`) rendered as `**Label:** value` lines, one blank line between (`jobs/_currently.py`); a legacy verbatim `currently.md` is still accepted.
- `haiku.md` — `compose-haiku` writes (rendered bold + hard-break by `_base.format_haiku` at draft/publish time); `metadata.json` — `compose-meta` writes; `cta-1.md` / `cta-2.md` — `compose-cta` writes.
- `draft.md` — `update-draft` writes (a regenerable pure projection of all the above + upstream, shaped like a delivered issue).
- `final.md` — `create-final` writes (post-Eddy ordering; still block-structured).
- `publish.md` — `build-publish` writes (the email body, byte-shaped like a raw Buttondown body: leading editor-mode comment, `---`-fenced parts intro → `## Currently` → cover → sections → membership-block-wrapped CTAs → haiku close → email-only tinylytics pixel; no block markers; the artifact `pipeline/content/content.py publish --issue N` pushes to Buttondown as a draft). `publish.html` alongside it is a Liquid-stripped preview.
- `cover.jpg`, `cover-large.jpg`, `journal/*.jpg`, `body-{N}.mp3`, `weekly-thing-{N}.mp3` — written by the iOS Shortcuts and `pipeline/audio/`; agents can't touch them (the `tools/s3.py` extension allowlist is text-only). `eddy-edits.md` — Eddy, rarely, for a preserved revision proposal.

**Issue-markdown shape (mirrors the iOS-Shortcut bodies in `data/buttondown/bodies/`).** Top-level blocks are `---`-fenced, in order: intro prose → `## Currently` (dropped when empty) → cover (`![](.../cover.jpg)` + caption + `Month D, YYYY  \nLocation`) → `## Notable` → `## Journal` → `## Briefly` → `A haiku to leave you with…` + the haiku + the closing "discuss on Reddit" line + `👨‍💻`. The list loops have strict formatting:
- **Notable** — leads with `_You can discuss any of these links at the [Weekly Thing {N} tag in r/WeeklyThing](…flair_name%3A%22Weekly%20Thing%20{N}%22)._` (carries the issue number — generated, not templated), one blank line, then items. Each item is `### [Title](url)` then `\n\n{commentary}`. **Two blank lines between items.**
- **Briefly** — each item is `{commentary} → **[Title](url)**` (commentary first, then the `→` arrow, then the bolded link; no headings). **One blank line between items.**
- **Journal** — a status update (no title) is `[Weekday @ H:MM AM/PM](url)\n\n{content}` (e.g. `[Sunday @ 4:16 PM](url)`); a *titled* post is "elevated": `### [Title](url)  \n{Weekday @ H:MM AM/PM}\n\n{content}` (H3 link with a markdown hard break, label plain on the next line). The label is weekday-name + 12-hour clock — no date — because every entry sits inside the same seven-day issue window, so the weekday already identifies it. Photos already live inside `content_md` (rehosted, emitted as native `<img>` tags), each on its own paragraph. **Two blank lines between entries.**
- **Haiku** — `**line one  \nline two  \nline three**` (bold-wrapped, markdown hard break between lines).

Renderers live in `jobs/update_draft.py` (`_render_notable` / `_render_brief` / `_render_journal`, `_reddit_tag_line`); `_base.format_haiku` is shared with `build-publish`. The membership CTA, the editor-mode comment, and the email tracking pixel are *not* in `draft.md` — they only appear in `publish.md` (the CTA composed by `compose-cta` and wrapped in the membership-block Liquid scaffold by `build-publish`; the editor-mode comment + tinylytics pixel added by `build-publish`). `update-draft` rebuilds `draft.md` from `templates/draft_starter.md` every run, so the layout always matches the template.

`tools/draft.py` (`section_status`) parses the draft for section item counts + asset presence + what's still missing for ship; `draft__section_status` is the agent tool over it. `tools/context.py` builds the per-persona dynamic-context blocks (`build_eddy_context` / `build_linky_context` / `build_patty_context` / `build_marky_context`) — runtime-computed facts injected into job prompts so the model doesn't recompute date math, word counts, queue depths, goal progress.

`tools/interaction.py` is the reaction primitive (`await_choice` / `await_approval`) — post options, wait for Jamie's reaction (filtered on `DISCORD_OWNER_USER_ID`), return the pick. Jobs hold their asset lock for the duration of the interaction.

## Storage — two layers, hard boundary

- **Public S3** `s3://files.thingelstad.com/weekly-thing/{N}/` — only what ships (text/JSON via the `tools/s3.py` allowlist, plus binaries written by other pipelines). Backs the `workspace__*` tools (`workspace__list_all` / `list_files` / `read` / `write`). Alongside the per-issue folders sits **`weekly-thing/workshop.json`** — a single pointer file `start-issue` (re)writes each time the in-flight issue window is set: `{issue_number, pub_date, end_date, start_date, day_count, workspace_url, workspace_prefix, archive_url, reddit_tag_url, files: {cover_jpg, cover_json, intro_md, currently_json, haiku_md, metadata_json, draft_md, draft_html, final_md, publish_md, publish_html}, set_at, set_by}`. Jamie's iOS Shortcuts fetch `https://files.thingelstad.com/weekly-thing/workshop.json` to discover the current issue + the exact URLs to PUT assets to (no-cache + CloudFront invalidation, so Shortcuts always see the latest).
- **SQLite** `apps/workshop_bot/data/workshop.db` — everything else. Tables: `agent_outputs`, `agent_runs`, `agent_notes`, `issue_windows`, `link_candidates`, `pinboard_popular_seen`, `pinboard_research_done`, `subscriber_events_seen`, `thingy_tokens`, `thingy_requests`, `thingy_conversations` (operator's mirror of reader↔Thingy conversations — `thingy-watch` writes; `/workshop thingy recent`/`show` read), `follow_ups` (agent/Jamie commitments — `followup__schedule` / `/workshop followup add` write; `follow-up-sweep` fires); **new in the content-loop redesign:** `job_locks`, `draft_digests`, `goals`, `campaigns`, `campaign_metrics`.

There is **no private workshop bucket** anymore. Decommissioned in the redesign: `s3://weekly-thing-workshop/`, the `WORKSHOP_BUCKET` env var, `tools/persona_s3.py` + the `s3_personas__*` tools, `tools/inbox.py` + the `agent_inbox` table + the `inbox__*` tools, the reserved-but-never-wired `analytics` / `supporter_events` / `channel_routes` tables, the `/workshop heartbeat` and `/workshop next-issue` slash commands. `s3_issues__*` was renamed to `workspace__*`. `db/schema.sql` carries a `DROP TABLE IF EXISTS` for each dropped table so a long-lived DB converges with a fresh install.

## Conventions worth remembering

- **Pinboard tag convention is strict: only `_brief`.** Among the bookmarks `update-draft` projects into the issue: untagged-in-window = Notable; `_brief` = Briefly. The old `_featured` tag/section is retired. `client.capture_blurb` writes a blurb verbatim + tags `_brief` + clears `toread` atomically (still available for direct use). **`client.issue_window_candidates` only returns *public, read* bookmarks** — anything still flagged `toread` (Jamie hasn't worked through it) or marked private (`shared=no`) is excluded, so the draft never picks up an unreviewed link. (Linky's `pinboard-scan` still *sees* the `toread` pile — that's its job; the exclusion is only in the draft projection.) Linky's reply-to-card flow is the soft sibling: Jamie's Discord reply seeds the bookmark's description without tagging `_brief` or clearing `toread`, so he can finish curation in Pinboard.
- **`pinboard__popular_unseen` filters avoid-domains.** Before the `pinboard_popular_seen` dedup it drops items whose host is on `tools/avoid_domains.py`'s exclusion set (own domains, Buttondown, image CDNs, URL shorteners, social, Wikipedia, YouTube/Spotify/Vimeo, generic CDNs) — so Linky never even considers a "popular on Pinboard" Wikipedia article or t.co redirect as a possible Notable. That list is a hand-maintained *copy* of `pipeline/content/domain_exclusions.py` (`pipeline/` isn't importable from here); keep them loosely in sync — an exact match isn't required since this only gates Linky's scan, nothing it touches ships.
- **micro.blog: pull everything in window, no tag filtering** (`tools/microblog.py`). Uses the Micropub source query (`GET {MICROBLOG_MICROPUB_URL}?q=source`, `Authorization: Bearer MICROBLOG_API_KEY`) — returns posts as mf2-JSON with `properties.content` carrying the **native markdown Jamie wrote** (a string; micro.blog embeds photo uploads as `<img src=…>` tags inside it). `MICROBLOG_API_KEY` is **required** — no fallback; if micro.blog is down, `journal.fill` leaves a placeholder line (Eddy's review flags it). Filtering/curation happens at `create-final` (Eddy can cut Journal entries).
- **Journal images are rehosted and emitted as native `<img>` tags** (`tools/journal_images.py`, called from `_gather_fills`). Each `<img>` / `![]()` on an upload host (`MICROBLOG_IMAGE_HOSTS`, default `www.thingelstad.com` `/uploads/`, `micro.thingelstad.com`, `cdn.uploads.micro.blog`, `uploads.micro.blog`) is downloaded, resized to `MICROBLOG_IMAGE_MAX_DIM` (default 600px long side, downscale-only; JPEG q80 / PNG optimized), and copied into `weekly-thing/{N}/journal/<name>` (reusing the micro.blog hash basename), then the reference is rewritten to the local URL. Every output image — whether the source was a markdown `![]()` or a `<img>` tag — is emitted as a native `<img src="…" alt="…" />` with width/height/etc. preserved, each on its own paragraph (so adjacent gallery `<img><img>` don't run together). Going forward we use `<img>` for images everywhere we generate them (cover + journal), because it gives the alt attribute an explicit slot and leaves room for `width`/`height`/`loading`/`class` later; Eleventy and Buttondown both pass raw HTML through. After rehost/normalize, rehosted images with empty alts are filled via `tools.alt_text` (vision LLM, cached forever per content-addressed filename; per-run cap of 15 calls). A HEAD-skip on rehost + the alt cache make daily re-runs cheap; a per-image failure leaves that one's original URL alone, and an alt failure leaves the alt empty (the hygiene review picks it up). Binary writes go through `s3.write_journal_image` (image-only allowlist, `journal/` sub-prefix; *not* an agent tool). `update-draft` runs `_gather_fills` (the blocking source pulls + image rehosting + the HTML-preview write) via `asyncio.to_thread`.
- **Browser-viewable HTML previews.** `update-draft` / `create-final` / `build-publish` write a `.html` twin alongside their `.md` (`draft.html` / `final.html` / `publish.html` in the workspace) via `tools.render` (Python-Markdown → a self-contained styled page; the CSS mirrors `content/buttondown/newsletter/buttondown-email.css`; `draft`/`final` strip the block markers and get a "DRAFT…/FINAL…" banner, `publish` renders clean). They're uploaded with `Cache-Control: no-cache` and a CloudFront invalidation of that path (`tools.cdn.invalidate`, distribution `WEEKLY_THING_CDN_DISTRIBUTION_ID`, default the prod one; set to empty to disable) — so Jamie always sees the latest at `https://files.thingelstad.com/weekly-thing/{N}/draft.html` etc. The URL is surfaced in the job's result and (for `update-draft`, Tue–Fri) appended to Eddy's review card. Best-effort: a render/invalidation hiccup never fails the job. The pages carry a `<meta name="robots" content="noindex,nofollow">` and `files.thingelstad.com/robots.txt` is already `Disallow: /`, so the in-progress drafts aren't indexed (they *are* publicly reachable by URL — same as `draft.md` always has been on that public prefix).
- **The draft.html editorial-review drawer.** When `update-draft` produces a review (`prompts/eddy/draft-review.md` — a solid pass: anchored, concrete *suggestions only*, never a rewrite — run via `_draft_review` on sonnet, every run), `render.render_and_upload_html(..., review_md=…)` injects a fixed "Show review" button and a slide-in drawer on the right holding the rendered review. The drawer is **hidden by default** (`transform: translateX(100%)`, revealed by `body.rv-open` via a tiny inline `<script>` toggle) — so the same shareable URL reads as the clean draft until someone flips it on. When it's open and the viewport is wide enough (`min-width: 880px`), `body.rv-open` also shifts the draft column left (extra right margin reserved for the panel) so the review and the content sit side by side instead of the panel covering the text; on narrower windows the panel overlays. No review (empty draft → Eddy `PASS`, or no team) → no button, no drawer. The review prompt walks Intro → Notable → Journal → Briefly → Whole issue → **Hygiene** — that last pass is a deliverability-and-reader-skepticism lens (anchor text quality, anchor/domain mismatch, hype headings, spam-filter signal in voice, alt-text quality, subject-shaping section headings) folded into the same single Sonnet call, no separate scanner. The Discord `#editorial` card (`update-review.md`, Tue–Fri) is separate and unchanged — different audience, different cadence.
- **Off-loop source pulls.** Jobs that hit external APIs from inside the bot's asyncio loop wrap the blocking calls in `asyncio.to_thread` so a slow Pinboard/Buttondown/Tinylytics/RSS/S3 round trip doesn't stall the gateway: `update-draft` (`_gather_fills` — Pinboard + micro.blog + image rehost + HTML render), `pinboard-scan` (`build_linky_context`), `daily-metrics` (campaign polls + `subscriber_growth` + `tinylytics.summary` + `build_marky_context`), `promotion-prep` (RSS lookup + `publish.md` read + `build_marky_context`). Agent-loop calls (`bot.core`) are already thread-wrapped in `agent_loop.py`.
- **Marky operates on the last *published* issue**, not the in-flight one — the RSS feed (`tools/rss.py`) is her trigger; she reads `publish.md`.
- **`update-draft` is a pure projection** — never an additive merge. Real authoring lives upstream. Re-running it gives the same output modulo upstream changes.
- **Job prompts get a `## Today` dynamic-context block.** Read it; don't recompute date math / counts / queue depth / goal progress.
- Prompt name → file: `team` → `prompts/shared/team.md`; `<persona>` → `prompts/<persona>/prompt.md`; `<persona>-<file>` → `prompts/<persona>/<file>.md` (e.g. `eddy-update-review`, `linky-research-card`, `marky-promotion-prep`, `patty-compose-cta`). Edits need a bot restart (prompts are cached at first read).
- New schema → `db/schema.sql` (idempotent — re-runs every boot; the `goals` seed uses `WHERE NOT EXISTS`); new *columns* on existing tables → `db._COLUMN_MIGRATIONS`.
- Tests: `python -m unittest discover -s apps/workshop_bot/tests -t .` — discord/anthropic/httpx stubbed (`tests/_stubs.py`); S3/Pinboard/micro.blog/Anthropic stubbed per-test. ~300 tests. Don't hit the network in tests.
