# workshop_bot ŌĆö project memory

> Author-only Discord bot for *The Weekly Thing*. Helps Jamie assemble each week's issue and ships supporting bits (link curation, membership CTA, promotion drafts). One Python process, several `discord.py` clients (one per persona, own token/avatar), plus an APScheduler instance ŌĆö all in the same asyncio loop.

Full design rationale: [`docs/workshop-content-loop-design-brief.md`](../../docs/workshop-content-loop-design-brief.md). The user-facing overview is [`README.md`](README.md). This file is the "what to keep in mind when working here" memory.

## Architecture: jobs are the spine

Every workshop_bot action ŌĆö pulling content into the draft, reordering for the final, composing CTAs, drafting promotion copy ŌĆö is a **job**: deterministic Python in `apps/workshop_bot/jobs/`, fired by the `/workshop ŌĆ”` slash surface (host: Eddy) and/or by cron. Some jobs make small encapsulated LLM calls (Eddy's draft review, the compose-* jobs, Marky's reports); most are pure Python. The job is the unit of scheduling and on-demand execution.

The slash surface is organised around the *content artifact*, not around the word "job": `/workshop issue {start,update,status,final,haiku,subject,cta,publish}`, `/workshop links scan`, `/workshop promo {prep,metrics}`, `/workshop campaign {add,edit,report,copy,sunset}`, `/workshop goal {set,done}`, `/workshop thingy {recent,show,sync}`, plus a top-level `/workshop status`. The internal job names used below (and by the scheduler, the `job_locks` table, and `agent_runs`) are the dash-cased ones ŌĆö `update-draft`, `create-final`, `compose-haiku`/`-meta`/`-cta`, `build-publish`, `pinboard-scan`, `promotion-prep`, `daily-metrics`, `thingy-watch`, etc.; the slash names are the operator-facing aliases (`/workshop issue update` ŌåÆ `update-draft`, `/workshop issue subject` ŌåÆ `compose-meta`, `/workshop links scan` ŌåÆ `pinboard-scan`, `/workshop promo metrics` ŌåÆ `daily-metrics`, `/workshop goal done` ŌåÆ `goal-achieved`, `/workshop thingy sync` ŌåÆ `thingy-watch`, and so on).

`jobs/_base.py` is the runtime: `JobContext` (carries `deps` ŌĆö corpus/registry/team ŌĆö and a `trigger` label; has `channel()` / `post()` helpers), `JobResult` (`{ok, message, data}` ŌĆö `message` is acked to the invoker), single-asset locking (`job_lock([assets], name)` context manager backed by the `job_locks` SQLite table ŌĆö two jobs that write the same file can't overlap; a lock held by a dead pid is stolen since a restart gets a new pid), and the draft-block helpers (`replace_block` / `get_block` for the `<!-- block:NAME -->` ŌĆ” `<!-- /block:NAME -->` markers in `draft.md`).

### The unified asset pattern

Every piece of issue content is a standalone file in the S3 workspace `s3://files.thingelstad.com/weekly-thing/{N}/`:

- `intro.md` (the opener prose) ŌĆö Jamie writes (Drafts ŌåÆ Shortcut); `update-draft` reads it verbatim into its block. The cover block's caption/date/location comes from `cover.json` ŌĆö `{"caption":ŌĆ”,"location":ŌĆ”,"timestamp":ŌĆ”}` rendered as `caption\n\ntimestamp  \nlocation` with the derived `![](.../cover.jpg)` prepended (`jobs/_cover.py`); a legacy verbatim `cover.md` (caption + `date  \nlocation`) is still accepted. The optional "Currently" section comes from `currently.json` ŌĆö a `{"Label":"value",ŌĆ”}` object (e.g. `{"Listening":"ŌĆ”","Watching":"ŌĆ”"}`) rendered as `**Label:** value` lines, one blank line between (`jobs/_currently.py`); a legacy verbatim `currently.md` is still accepted.
- `haiku.md` ŌĆö `compose-haiku` writes (rendered bold + hard-break by `_base.format_haiku` at draft/publish time); `metadata.json` ŌĆö `compose-meta` writes; `cta-1.md` / `cta-2.md` ŌĆö `compose-cta` writes.
- `draft.md` ŌĆö `update-draft` writes (a regenerable pure projection of all the above + upstream, shaped like a delivered issue).
- `final.md` ŌĆö `create-final` writes (post-Eddy ordering; still block-structured).
- `publish.md` ŌĆö `build-publish` writes (the email body, byte-shaped like a raw Buttondown body: leading editor-mode comment, `---`-fenced parts intro ŌåÆ `## Currently` ŌåÆ cover ŌåÆ sections ŌåÆ membership-block-wrapped CTAs ŌåÆ haiku close ŌåÆ email-only tinylytics pixel; no block markers; the artifact `pipeline/content/content.py publish --issue N` pushes to Buttondown as a draft). `publish.html` alongside it is a Liquid-stripped preview.
- `cover.jpg`, `cover-large.jpg`, `journal/*.jpg`, `body-{N}.mp3`, `weekly-thing-{N}.mp3` ŌĆö written by the iOS Shortcuts and `pipeline/audio/`; agents can't touch them (the `tools/s3.py` extension allowlist is text-only). `eddy-edits.md` ŌĆö Eddy, rarely, for a preserved revision proposal.

**Issue-markdown shape (mirrors the iOS-Shortcut bodies in `data/buttondown/bodies/`).** Top-level blocks are `---`-fenced, in order: intro prose ŌåÆ `## Currently` (dropped when empty) ŌåÆ cover (`![](.../cover.jpg)` + caption + `Month D, YYYY  \nLocation`) ŌåÆ `## Notable` ŌåÆ `## Journal` ŌåÆ `## Briefly` ŌåÆ `A haiku to leave you withŌĆ”` + the haiku + the closing "discuss on Reddit" line + `­¤æ©ŌĆŹ­¤Æ╗`. The list loops have strict formatting:
- **Notable** ŌĆö leads with `_You can discuss any of these links at the [Weekly Thing {N} tag in r/WeeklyThing](ŌĆ”flair_name%3A%22Weekly%20Thing%20{N}%22)._` (carries the issue number ŌĆö generated, not templated), one blank line, then items. Each item is `### [Title](url)` then `\n\n{commentary}`. **Two blank lines between items.**
- **Briefly** ŌĆö each item is `{commentary} ŌåÆ **[Title](url)**` (commentary first, then the `ŌåÆ` arrow, then the bolded link; no headings). **One blank line between items.**
- **Journal** ŌĆö a status update (no title) is `[Mon D, YYYY at H:MM AM/PM](url)\n\n{content}`; a *titled* post is "elevated": `### [Title](url)  \n{Mon D, YYYY at H:MM AM/PM}\n\n{content}` (H3 link with a markdown hard break, date plain on the next line). Photos already live inside `content_md` (rehosted), each on its own paragraph. **Two blank lines between entries.**
- **Haiku** ŌĆö `**line one  \nline two  \nline three**` (bold-wrapped, markdown hard break between lines).

Renderers live in `jobs/update_draft.py` (`_render_notable` / `_render_brief` / `_render_journal`, `_reddit_tag_line`); `_base.format_haiku` is shared with `build-publish`. The membership CTA, the editor-mode comment, and the email tracking pixel are *not* in `draft.md` ŌĆö they only appear in `publish.md` (the CTA composed by `compose-cta` and wrapped in the membership-block Liquid scaffold by `build-publish`; the editor-mode comment + tinylytics pixel added by `build-publish`). `update-draft` rebuilds `draft.md` from `templates/draft_starter.md` every run, so the layout always matches the template.

`tools/draft.py` (`section_status`) parses the draft for section item counts + asset presence + what's still missing for ship; `draft__section_status` is the agent tool over it. `tools/context.py` builds the per-persona dynamic-context blocks (`build_eddy_context` / `build_linky_context` / `build_patty_context` / `build_marky_context`) ŌĆö runtime-computed facts injected into job prompts so the model doesn't recompute date math, word counts, queue depths, goal progress.

`tools/interaction.py` is the reaction primitive (`await_choice` / `await_approval`) ŌĆö post options, wait for Jamie's reaction (filtered on `DISCORD_OWNER_USER_ID`), return the pick. Jobs hold their asset lock for the duration of the interaction.

## Conventions worth remembering

- **Pinboard tag convention is strict: only `_brief`.** Among the bookmarks `update-draft` projects into the issue: untagged-in-window = Notable; `_brief` = Briefly. The old `_featured` tag/section is retired. `pinboard__capture_blurb` writes a blurb verbatim + tags `_brief` + clears `toread` atomically. **`client.issue_window_candidates` only returns *public, read* bookmarks** ŌĆö anything still flagged `toread` (Jamie hasn't worked through it) or marked private (`shared=no`) is excluded, so the draft never picks up an unreviewed link. (Linky's `pinboard-scan` still *sees* the `toread` pile ŌĆö that's its job; the exclusion is only in the draft projection.)
- **`pinboard__popular_unseen` filters avoid-domains.** Before the `pinboard_popular_seen` dedup it drops items whose host is on `tools/avoid_domains.py`'s exclusion set (own domains, Buttondown, image CDNs, URL shorteners, social, Wikipedia, YouTube/Spotify/Vimeo, generic CDNs) ŌĆö so Linky never even considers a "popular on Pinboard" Wikipedia article or t.co redirect as a possible Notable. That list is a hand-maintained *copy* of `pipeline/content/domain_exclusions.py` (`pipeline/` isn't importable from here); keep them loosely in sync ŌĆö an exact match isn't required since this only gates Linky's scan, nothing it touches ships.
- **micro.blog: pull everything in window, no tag filtering** (`tools/microblog.py`). Uses the Micropub source query (`GET {MICROBLOG_MICROPUB_URL}?q=source`, `Authorization: Bearer MICROBLOG_API_KEY`) ŌĆö returns posts as mf2-JSON with `properties.content` carrying the **native markdown Jamie wrote** (a string; micro.blog embeds photo uploads as `<img src=ŌĆ”>` tags inside it). `MICROBLOG_API_KEY` is **required** ŌĆö no fallback; if micro.blog is down, `journal.fill` leaves a placeholder line (Eddy's review flags it). Filtering/curation happens at `create-final` (Eddy can cut Journal entries).
- **Journal images are rehosted** (`tools/journal_images.py`, called from `journal.fill`). Each `<img>` / `![]()` on an upload host (`MICROBLOG_IMAGE_HOSTS`, default `www.thingelstad.com` `/uploads/`, `micro.thingelstad.com`, `cdn.uploads.micro.blog`, `uploads.micro.blog`) is downloaded, resized to `MICROBLOG_IMAGE_MAX_DIM` (default 600px long side, downscale-only; JPEG q80 / PNG optimized), and copied into `weekly-thing/{N}/journal/<name>` (reusing the micro.blog hash basename), then the reference is rewritten to the local URL and `<img>` tags are normalized to `![alt](url)`, each on its own paragraph (so adjacent gallery `<img><img>` don't run together). A HEAD-skip makes daily re-runs cheap; a per-image failure leaves that one's original URL alone. Binary writes go through `s3.write_journal_image` (image-only allowlist, `journal/` sub-prefix; *not* an agent tool). `update-draft` runs `_gather_fills` (the blocking source pulls + image rehosting + the HTML-preview write) via `asyncio.to_thread`.
- **Browser-viewable HTML previews.** `update-draft` / `create-final` / `build-publish` write a `.html` twin alongside their `.md` (`draft.html` / `final.html` / `publish.html` in the workspace) via `tools.render` (Python-Markdown ŌåÆ a self-contained styled page; the CSS mirrors `content/buttondown/newsletter/buttondown-email.css`; `draft`/`final` strip the block markers and get a "DRAFTŌĆ”/FINALŌĆ”" banner, `publish` renders clean). They're uploaded with `Cache-Control: no-cache` and a CloudFront invalidation of that path (`tools.cdn.invalidate`, distribution `WEEKLY_THING_CDN_DISTRIBUTION_ID`, default the prod one; set to empty to disable) ŌĆö so Jamie always sees the latest at `https://files.thingelstad.com/weekly-thing/{N}/draft.html` etc. The URL is surfaced in the job's result and (for `update-draft`, TueŌĆōFri) appended to Eddy's review card. Best-effort: a render/invalidation hiccup never fails the job. The pages carry a `<meta name="robots" content="noindex,nofollow">` and `files.thingelstad.com/robots.txt` is already `Disallow: /`, so the in-progress drafts aren't indexed (they *are* publicly reachable by URL ŌĆö same as `draft.md` always has been on that public prefix).
- **The draft.html editorial-review drawer.** When `update-draft` produces a review (`prompts/eddy/draft-review.md` ŌĆö a solid pass: anchored, concrete *suggestions only*, never a rewrite ŌĆö run via `_draft_review` on sonnet, every run), `render.render_and_upload_html(..., review_md=ŌĆ”)` injects a fixed "Show review" button and a slide-in drawer on the right holding the rendered review. The drawer is **hidden by default** (`transform: translateX(100%)`, revealed by `body.rv-open` via a tiny inline `<script>` toggle) ŌĆö so the same shareable URL reads as the clean draft until someone flips it on. When it's open and the viewport is wide enough (`min-width: 880px`), `body.rv-open` also shifts the draft column left (extra right margin reserved for the panel) so the review and the content sit side by side instead of the panel covering the text; on narrower windows the panel overlays. No review (empty draft ŌåÆ Eddy `PASS`, or no team) ŌåÆ no button, no drawer. The Discord `#editorial` card (`update-review.md`, TueŌĆōFri) is separate and unchanged ŌĆö different audience, different cadence.
