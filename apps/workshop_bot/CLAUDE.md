# workshop_bot — project memory

> Author-only Discord bot for *The Weekly Thing*. Helps Jamie assemble each week's issue and ships supporting bits (link curation, membership CTA, promotion drafts). One Python process, several `discord.py` clients (one per persona, own token/avatar), plus an APScheduler instance — all in the same asyncio loop.

Full design rationale: [`docs/workshop-content-loop-design-brief.md`](../../docs/workshop-content-loop-design-brief.md). The user-facing overview is [`README.md`](README.md). This file is the "what to keep in mind when working here" memory.

## Architecture: jobs are the spine

Every workshop_bot action — pulling content into the draft, reordering for the final, composing CTAs, drafting promotion copy — is a **job**: deterministic Python in `apps/workshop_bot/jobs/`, fired by a per-persona slash tree and/or by cron. Some jobs make small encapsulated LLM calls (Eddy's draft review, the compose-* jobs, Marky's reports); most are pure Python. The job is the unit of scheduling and on-demand execution.

Each persona hosts its own slash tree on its own Discord bot (no single `/workshop` umbrella anymore):

- `/eddy issue {start,update,status,final,haiku,subject,publish,send,reset}` · `/eddy edit <asset>` · `/eddy currently {list,edit,set,clear,reorder,add-type,retire-type}` · `/eddy status` · `/eddy review <text>` · `/eddy archive <issue>` · `/eddy followup {list,add,cancel}`
- `/linky scan` · `/linky research <url>` · `/linky pile` · `/linky stats` · `/linky followup {list,add,cancel}`
- `/marky prep` · `/marky metrics` · `/marky engagement` · `/marky referrers` · `/marky campaign {add,edit,report,copy,sunset}` · `/marky followup {list,add,cancel}`
- `/patty cta` · `/patty goal {set,done}` · `/patty progress` · `/patty nonprofit` · `/patty supporters` · `/patty followup {list,add,cancel}`

The internal job names used below (and by the scheduler, the `job_locks` table, and `agent_runs`) are the dash-cased ones — `update-draft`, `create-final`, `compose-haiku`/`-meta`/`-cta`, `build-publish`, `send-to-buttondown`, `reset-issue`, `edit-asset`, `pinboard-scan`, `promotion-prep`, `daily-metrics`, `follow-up-sweep`, etc.; the slash names are the operator-facing aliases (`/eddy issue update` → `update-draft`, `/eddy issue subject` → `compose-meta`, `/eddy issue send` → `send-to-buttondown`, `/eddy issue reset` → `reset-issue`, `/eddy edit` → `edit-asset`, `/linky scan` → `pinboard-scan`, `/marky metrics` → `daily-metrics`, `/patty goal done` → `goal-achieved`, and so on).

The reader-facing Thingy bot — the public Q&A surface in `#ask-thingy` plus the operator-side `thingy-watch` conversation mirror + `/thingy {recent,show,sync}` commands — lives in a separate process at [`../thingy_bridge/`](../thingy_bridge/). See that app's CLAUDE.md for its architecture.

`jobs/_base.py` is the runtime: `JobContext` (carries `deps` — corpus/registry/team — and a `trigger` label; has `channel()` / `post()` helpers), `JobResult` (`{ok, message, data}` — `message` is acked to the invoker), single-asset locking (`job_lock([assets], name)` context manager backed by the `job_locks` SQLite table — two jobs that write the same file can't overlap; a lock held by a dead pid is stolen since a restart gets a new pid), and the draft-block helpers (`replace_block` / `get_block` for the `<!-- block:NAME -->` … `<!-- /block:NAME -->` markers in `draft.md`).

### The jobs

| Job | Trigger | What it does |
|---|---|---|
| `start-issue <n> <pub-date> <days>` (`/eddy issue start`) | manual | Records the issue window in `workshop.db`, refreshes the `weekly-thing/workshop.json` pointer Shortcuts read (issue number, dates, predictable workspace URLs), seeds `draft.md` from `templates/draft_starter.md`, auto-fires `update-draft`. The only job that takes the issue number explicitly. |
| `update-draft` | daily 17:00 CT + after `start-issue` + manual | Pure projection: rebuilds `draft.md` from `templates/draft_starter.md` every run (so the layout always tracks the template) and fills each block. Notable/Briefly/Journal come from `issue_items` rows — `issue_items_sync.sync_all` first refreshes rows from Pinboard + micro.blog (UPSERT on stable source ids; non-promoted rows whose upstream item disappeared are pruned), then `issue_items_render` projects rows back to section markdown. Atoms stay file-backed: `intro.md` / `haiku.md` verbatim from the authored asset files, the cover caption/date/location from `cover.json` (`{"caption":…,"location":…,"timestamp":…,"alt":…}` rendered as `caption\n\ntimestamp  \nlocation`, native `<img alt="…" src=".../cover.jpg" />` prepended — preferred) or a legacy verbatim `cover.md` (`_cover.render` / `_cover.alt`). **`Currently` is now DB-backed** — `_currently.render` reads from `currently_entries` joined with `currently_types` (ordered by per-issue `position`); the retired iOS-Drafts → Shortcut → `currently.json` path is no longer consulted. Output is shaped like a delivered issue (see "issue-markdown shape" below): `---`-fenced blocks, the Notable "discuss on Reddit" line, `### [Title](url)` link headings, `→ **[Title](url)**` Briefly form, elevated (titled) Journal posts, the `A haiku to leave you with…` close. Idempotent; replaces block content wholesale, nothing on the old draft preserved. **Alt-text auto-fill:** any rehosted journal image *and* the cover image whose alt is missing/empty gets one filled via a Claude vision call (`tools/alt_text.py`, sonnet, ≤120 chars, factual, no "photo of" preamble); cached forever in `image_alt_cache` keyed by content-addressed filename (`428e3db12e.jpg`) / `cover-{N}`, so the daily re-run is free once each image is seen. `cover.json.alt`, if set by the operator, wins over the vision call. Per-run cap of 15 calls (`WORKSHOP_ALT_VISION_CAP`); cap-exhaustion or generator failure leaves alt empty — Eddy's Hygiene review surfaces those. Then: a **solid editorial review** of the draft (`prompts/eddy/draft-review.md`, sonnet, every run) gets embedded in `draft.html` behind a "Show review" toggle — hidden by default, so the shareable link reads clean until someone flips it on (`_draft_review` → `render.render_and_upload_html(review_md=…)`); each target-anchored bullet in the review is captured to `editorial_comments` with a stable handle (`E349-N1`, `E349-X3`, etc.) and rendered as a copy-button badge in the drawer so Jamie can paste the handle into Discord to continue the conversation (`@eddy tell me about E349-N1` resolves via the `editorial__get_comment` tool); writes a `draft_digests` row; on Tue–Fri also runs Eddy's `#editorial` post-update card (`prompts/eddy/update-review.md`), silent Sat/Sun/Mon. The review prompt branches on `review_tier` (`early` / `mid` / `ship_eve` — derived from days-to-publish + iteration count + open-comments count) so feedback adapts to where in the cycle we are. Refuses (loudly) if `final.md` exists — the issue is locked (use `/eddy issue reset final` to drop the gate). |
| `issue-status` | manual | Read-only section + asset completeness report. |
| `create-final` | manual | Eddy's editorial pass. **The LLM specifies the order; code mutates rows.** Reads `issue_items` rows (Notable / Brief / Journal — `tools/issue_items.py`), surfaces them to Eddy with stable synthetic ids (`n1`/`b2`/`j3`), asks for an ordering-only JSON proposal (`{thesis, *_order, promotions, membership_blocks}`), validates strictly (each `*_order` is a strict permutation of non-promoted ids; membership-block `after` refs can't target promoted ids), then applies: `clear_promotions` → `promote(row_id, ...)` per promotion → `reorder(issue, section, ordered_row_ids)` per section. **Promotions are Journal-only** — only Jamie's own journal entries (typically longer-form blog posts) can be elevated to a standalone featured section; Notable links and Brief items stay in their parent sections. The row model guarantees losslessness by construction (an UPDATE on position can't drop, duplicate, or mutate a row), so the old chunk multiset check is gone. Writes `final.md` (rendered from rows via `tools/issue_assembly.assemble_final` — promoted items splice inline as `## Heading` sections at their declared `promoted_position`, NOT in feature1/feature2 blocks at the bottom — and inline `<!-- cta:N -->` / `<!-- thanks:N -->` markers) and `thesis.md`. Also writes a **side-by-side `final-proposal.html`** (current vs proposed columns, SVG connector lines between matching items, promoted items struck-through on the left + rendered as a featured section on the right, membership markers as inline pills) so Jamie can read the editorial pass visually before reacting. Posts to `#editorial`: thesis + per-section was/now reorder map + promotions plan + membership-block plan + 📄 link to the side-by-side view; Jamie ✅/❌/🔄. On ❌: writes `final.md` from current row order, no thesis, no markers, no promotions. **Auto-fire on ✅:** when the approved proposal declared any cta/thanks slots, `compose-cta` is scheduled as a background task (no manual `/patty cta` step required — closes the WT348 skip-Patty failure mode). |
| `compose-haiku` | manual | Eddy returns JSON haiku options → `#editorial`; Jamie picks → `haiku.md`. The pick prompt also links a `haiku-options.html` page (option cards with a copy button per haiku, mono font preserving the line breaks). Required for ship. |
| `compose-meta` | manual | Two LLM steps in `#editorial`. **Subject:** Eddy returns **5 `WT<N> — <theme>` options** (`prompts/eddy/compose-subject.md` — Jamie's prompt, verbatim) → Jamie reacts 1️⃣–5️⃣ (🔄 = refresh). The pick prompt also links a `subject-options.html` page (5 cards with a copy button each, serif body) so Jamie can read the full lines without Discord wrapping them. **Description:** Eddy generates a single comma-separated topic line (`prompts/eddy/compose-description.md` — also Jamie's prompt, verbatim) — *no picker*, the prompt is deterministic enough (length window, ordering rules, style guardrails) for one-shot. The success post surfaces both so Jamie can spot a bad description and edit in Buttondown or re-run. Writes `metadata.json` (subject/description generated; image/slug/number/publish_date deterministic). Required for ship. |
| `compose-cta` | manual + auto-fired by `create-final` on ✅ | Patty fills the membership-block slots Eddy declared in `final.md`. Scans for `<!-- cta:N -->` and `<!-- thanks:N -->` markers, generates 1–2 framings per slot (kind-aware: supporter CTA in Thingy's voice for `cta:N`, sincere thank-you for `thanks:N`), Jamie picks per slot via the standard refresh-loop UX → writes `cta-N.md` / `thanks-N.md` with `kind:` YAML frontmatter (`supporter` or `thanks`). Each per-slot pick prompt also links a `{kind}-{slot_n}-options.html` cards page. Patty no longer decides count or placement — that's Eddy's call now. Already-filled slots are skipped; delete a file to re-roll (or use `/eddy edit cta-N` to revise in place). Thesis (`thesis.md`) injected into both prompts when present. Optional standalone; in practice it auto-fires after `create-final`. |
| `build-publish` | manual | Transforms `final.md` into `buttondown.md` — byte-shaped like a real Buttondown body. Reads `final.md` (already body-shaped after create-final, with promoted sections spliced inline and cta/thanks markers placed), strips block markers, substitutes each `<!-- cta:N -->` / `<!-- thanks:N -->` with an **audience-aware Liquid block** (`cta:N` shows the supporter ask + Stripe upsell to free subscribers and CTA + subscribe form to anonymous readers, premium falls through to nothing; `thanks:N` shows the thanks copy to premium members only), prepends `<!-- buttondown-editor-mode: plaintext -->`, and appends the email-only `{% if medium == 'email' %}…tinylytics pixel…{% endif %}`. Also writes `buttondown.html` — a Liquid-stripped regular-subscriber rendering (best-effort preview). Refuses with a missing-list (→ `#editorial`) if any required asset is absent (`final.md`, `haiku.md`, `metadata.json`, `intro.md`, `cover.jpg`) or if a marker declared in `final.md` has no copy file. |
| `send-to-buttondown` | manual | Pushes `buttondown.md` to Buttondown as a draft, **idempotently**. On the first run for an issue: POSTs `/emails`, captures the response `id`, writes it back to `metadata.json` as `buttondown_id`. On every subsequent run: PATCHes `/emails/{buttondown_id}` so re-running pushes edits to the same draft. If the draft was deleted in Buttondown's UI between runs (PATCH 404s), falls through to POST and overwrites the stored id. **`publish_date` is never sent** — the bot stages the draft; Jamie schedules, spot-checks, and ships from the Buttondown UI. Posts to `#editorial` with a "Created/Updated …" indicator + the [open in Buttondown] link. Wraps `pipeline/content/content.py:buttondown_publish_idempotent` so the CLI `pipeline/content/content.py publish --issue N` and the slash command share the same idempotent logic. `jobs/send_to_buttondown.py` + the slash command `/eddy issue send`. |
| `reset-issue {final\|publish}` (`/eddy issue reset`) | manual | Forces the in-flight issue back to an earlier step by deleting the gate artifacts. `reset final` deletes `final.md` / `thesis.md` / `final.html` and clears any `issue_items` promotions (so the next `/eddy issue final` proposes against the current row state). `reset publish` deletes `buttondown.md` / `buttondown.html`. Deliberately preserved: `issue_items` rows, `cta-N.md` / `thanks-N.md` files, and `metadata.json`'s `buttondown_id` (so the next `send-to-buttondown` PATCHes the same Buttondown draft rather than orphaning it with a fresh POST). Idempotent: a reset where nothing's present prints "nothing to reset". Each successful reset posts a one-line confirmation to `#editorial`. |
| `edit-asset <asset>` (`/eddy edit`) | manual | Pops a Discord modal pre-filled with the asset's current S3 contents; submit writes back. Asset is one of `intro`/`outro`/`haiku`/`cover`/`cta-1`/`cta-2`/`thanks-1`. For atoms that flow into `draft.md` (`intro`/`outro`/`haiku`/`cover`), the job also re-fires `update-draft` so the preview refreshes; `cta-N.md` / `thanks-N.md` edits don't auto-fire anything (those only affect `buttondown.md`, and that's a deliberate step). Discord modals cap at 4000 chars per `TextInput`, which fits every atom file; bodies over the cap refuse with a "use the S3 console" hint. Deliberately *not* editable here: `draft.md` / `final.md` / `buttondown.md` (renders, regenerate from atoms), `metadata.json` (structured; subject/description come from `compose-meta`), and `currently.json` (retired — the Currently section is DB-backed now; edit per-type via `/eddy currently edit <type>`). |
| `currently {list,edit,set,clear,reorder,add-type,retire-type}` (`/eddy currently`) | manual + Mon/Wed evening cron via `follow-up-sweep` | DB-backed `## Currently` editor. `list` shows the in-flight issue's filled entries + unfilled active types + stale-pick hint. `edit <type>` pops a per-type modal pre-filled with the current DB value (good for values carrying markdown links). `set` is the non-modal quick path; `clear` deletes one and renumbers; `reorder` takes a comma-separated permutation of filled labels. `add-type`/`retire-type` manage the canonical pool — adding "Printing" is just `/eddy currently add-type Printing`, no code change. Every mutating op refires `update-draft` so the preview refreshes. Eddy also handles this conversationally in `#editorial` via the `currently__*` agent tools (`set` / `clear` / `reorder` / `add_type` / `suggest_stale` / `list_types` / `list_entries`); Mon 17:00 CT + Wed 17:00 CT follow-ups (scheduled by `start-issue`) seed the week's conversation, asking Jamie about a stale type. The legacy iOS-Drafts → Shortcut → `currently.json` path is retired. |
| `pinboard-scan` | every 3h 07:00–22:00 CT (07/10/13/16/19/22), year-round; manual | Linky's per-link research pass. One **toread** lane (Jamie's own public-`toread` bookmarks not yet in `pinboard_research_done`, via `pinboard.toread_public_unresearched`) plus N **discovery feeds** declared in the `DISCOVERY_FEEDS` registry in `tools/feeds/feed_registry.py` — each entry is a `FeedSpec` carrying name, label, pin-label, fetch callable, per-scan cap, feed limit, and primary priority. Today's discovery set is **Pinboard popular only**; the registry remains generic so future feeds can be added intentionally, while retired non-Pinboard feed rows are removed by the DB migration. Cross-source mechanics still exist for the toread lane and any future feed: a **cross-day uplift** (URL was seen on a different lane/feed days ago → `## Cross-source uplift` block carries the prior sightings + original verdict; Linky reconsiders, capped at `_UPLIFT_PER_SCAN_CAP=5` per scan; sightings tracked in the `popular_seen_sightings` table). Uplift cards post as **loose `#research` messages** (no Discord reply-threading — that surface was removed in May 2026; the `## Cross-source uplift` context block still ships inside the card body). To add a feed: write `tools/feeds/<name>.py` with `top(limit=N)` (or equivalent) returning `[{url, title, [discussion_url], [score], [comment_count], [submitter], [tags]}]`, add one `FeedSpec` line, append the name to `db.RESEARCH_SOURCES`. The job (not the LLM) iterates candidates and calls Linky's `core` once per link with `prompts/linky/research-card.md` (source-agnostic — switches on `toread` vs. discovery via the `## The link` block's `Source` field); each per-link call returns either a Discord card or one of two signals — `SKIP: <reason>` (discovery sources only — judged not interesting; URL is marked seen and never resurfaces from the same feed, may still uplift if a *different* lane/feed surfaces it later) or `FETCH_FAILED: <reason>` (any source; URL is NOT marked, so it can retry next scan). Cards no longer include read-length/source footers or "also on" lines. Each card posts as its own `#research` message; the `discord_message_id` is recorded in `linky_research_messages` so a Discord reply / save-reaction can route back to the right Pinboard bookmark. URL normalisation for cross-source dedup is `tools/url_normalize.py`'s `dedup_key` (lowercases host, strips trailing slash, drops `utm_*`/`fbclid`/etc. — in-memory comparison only; stored URLs are unchanged). **Archive resonance** runs as a Python pre-step on every card: `ctx.deps.corpus.search(query, k=3)` against the in-memory BM25 archive corpus, results rendered as a `## Archive resonance` block (≤180-char snippets per hit) inside the `## The link` block. Query strategy is `title + description` for toread, `title` for discovery sources. The prompt explicitly says resonance is informational, not a filter — fresh territory doesn't need a higher bar, an echo doesn't earn a free pass. The LLM still has `archive__search` as a tool for follow-up phrasing if the obvious lookup misses. PASSes silently when all sources come back empty (common off-hour). |
| `promotion-prep` | auto on RSS detection; manual | Marky drafts syndication content (LinkedIn + r/WeeklyThing megathread + per-link threads, 2–3 framings each) for the latest *published* issue's `buttondown.md` → `#promotion`. Never auto-posts. |
| `daily-metrics` | daily 19:00 CT; manual | Polls active campaigns (Tinylytics `?ref=` traffic + Buttondown ref signups → a `campaign_metrics` row each run), checks subscriber growth + engagement; PASSes silently when nothing material moved, else Marky composes a terse report → `#promotion`. |

| `goal-achieved [notes]` | manual | Marks the active goal achieved (today); `notes` is appended to whatever was recorded at `set-goal` time. |
| `follow-up-sweep` | hourly (cron `23 * * * *`) | Fires due **follow-ups** — commitments an agent (or Jamie) made via `followup__schedule` / `/<persona> followup add`, stored in the `follow_ups` table. Two trigger kinds: time-based (`due_at` — any distance: tomorrow, next month, next year) and issue-based (`trigger_issue` — fires once the active in-flight issue reaches it). For each due one: runs the persona's agent loop with the note + a `## Today` context block, posts the agent's check-in to the persona's home channel, marks it fired. PASSes silently when nothing's due; an agent PASS marks it fired without posting; an agent/channel error leaves it open to retry; ≤8/sweep. `jobs/follow_up.py`. **This is the deliberate, targeted replacement for per-persona heartbeats** — it only does anything when a real commitment comes due. `jobs/follow_up.py` also backs each persona's `/<persona> followup {list,add,cancel}` slash subgroup and the `followup__schedule` / `followup__list` / `followup__cancel` agent tools. |

**Ad-hoc commands** added in the per-persona-tree rollout (each is a small handler in `jobs/<persona>_quicklook.py` or a small standalone module — none are scheduled):

- `/eddy review <text>` → `jobs/review_text.py` (ad-hoc editorial review of pasted text)
- `/eddy archive <issue>` → `jobs/archive_lookup.py` (past-issue overview)
- `/linky research <url>` → `jobs/linky_research.py` (ad-hoc per-URL research)
- `/linky pile` / `/linky stats` → `jobs/linky_quicklook.py`
- `/marky engagement` / `/marky referrers` → `jobs/marky_quicklook.py`
- `/patty progress` / `/patty nonprofit` / `/patty supporters` → `jobs/patty_quicklook.py`

Plus one **top-level cross-cutting** subcommand on Eddy: **`/eddy status`** — a read-only DB-only ops snapshot (active issue window, active goal/campaigns, any held `job_locks`, the last few `agent_runs`). Distinct from `/eddy issue status` (the in-flight issue's *content* completeness) and `/marky campaign report` (campaign perf vs expected). Source: `jobs/status.py`; `set-goal`/`goal-achieved`/`campaign-sunset`/`campaign-copy`/`campaign-edit` are in `jobs/ops.py` (tiny, no-LLM).

Scheduler: `scheduler/jobs.py` declares the cron `JobSpec`s; `scheduler/handlers.py` has `content_job` (the bridge cron→jobs, wired as `functools.partial(content_job, job="<name>")`) and `rss_check` (poll the feed, dedupe via `agent_notes`, auto-fire `promotion-prep`). There are **no per-persona heartbeats** — everything an agent does on a cadence is a job. The closest thing to "an agent acting on its own" is `follow-up-sweep` firing a commitment the agent itself scheduled.

Slash dispatch (`personas/commands/`): one register fn per persona — `register_{eddy,linky,marky,patty}_commands(bot)` — each builds and adds a top-level group to that persona's `CommandTree`. Shared dispatch helpers live in `personas/commands/_shared.py` (`make_ack`, `make_run_and_ack`, `make_run_interactive`, `_ctx`). The fast jobs `defer` → run → ack with the result (the ack send is wrapped in try/except — a Discord interaction token only lasts ~15 min). The **interactive** jobs (`create-final`, `compose-haiku`/`-meta`/`-cta` — they post options to a channel and wait for Jamie's reaction, which can outlast the token) instead ack *immediately* ("started — react in #editorial / #supporters") and then run; the job posts its own outcome to the channel, so no second followup is needed. `/eddy status` is attached directly under the `/eddy` group rather than under `/eddy issue …` (it's a bot-health view, not tied to a single artifact). Each persona's `PersonaBot.__init__` calls its register fn and the base class syncs the tree on `on_ready`.

### The unified asset pattern

Every piece of issue content is a standalone file in the S3 workspace `s3://files.thingelstad.com/weekly-thing/{N}/`:

- `intro.md` (the opener prose) and `outro.md` (the closing prose) — both Jamie-authored, pushed via Shortcut; `update-draft` reads each verbatim into its block (`intro` / `outro`). The cover block's caption/date/location comes from `cover.json` — `{"caption":…,"location":…,"timestamp":…}` rendered as `caption\n\ntimestamp  \nlocation` with the derived `![](.../cover.jpg)` prepended (`jobs/_cover.py`); a legacy verbatim `cover.md` (caption + `date  \nlocation`) is still accepted. The optional "Currently" section is **DB-backed** — one row per type per issue in `currently_entries`, ordered by `position` (insertion-order by default; reorderable by Eddy/Jamie). `jobs/_currently.render` reads from the DB and emits `**Label:** value` lines, one blank line between. The canonical type pool lives in `currently_types` (seed: Listening, Watching, Reading, Playing, Installing, Dining, Cooking, Making, Drinking, Printing). The retired iOS-Drafts → Shortcut → `currently.json` path is no longer read.
- `haiku.md` — `compose-haiku` writes (rendered bold + hard-break by `_base.format_haiku` at draft/publish time); `metadata.json` — `compose-meta` writes (subject/description/image/slug/publish_date); `send-to-buttondown` writes back the `buttondown_id` field on the first successful POST so re-runs PATCH the same draft. `cta-N.md` / `thanks-N.md` — `compose-cta` writes (1–2 supporter CTAs + 0–1 thank-you, each with `kind: supporter` / `kind: thanks` YAML frontmatter; the audience-aware Liquid wrapping happens later in `build-publish`).
- `thesis.md` — `create-final` writes on ✅ (one to three sentences in Eddy's voice: the editorial intent for the issue). Read by `compose-meta`, `compose-haiku`, and `compose-cta` so the subject options, description, haiku, and CTA framings all anchor on the same narrative.
- `draft.md` — `update-draft` writes (a regenerable pure projection of all the above + upstream, shaped like a delivered issue).
- `final.md` — `create-final` writes. Block-structured like the draft, with the Notable/Journal/Brief blocks rendered from `issue_items` rows in the reordered position (`tools/issue_items.py` + `tools/issue_items_render.py`); the row model guarantees losslessness by construction. Inline `<!-- cta:N -->` / `<!-- thanks:N -->` markers sit at the positions Eddy declared. **Promoted items splice inline as standalone `## {heading}` sections at their declared `promoted_position`** (between Notable and Journal, or between Journal and Briefly, etc.) — what Jamie reads in `final.md` is exactly where the promoted section will appear in the published email. No `feature1` / `feature2` blocks at the bottom of the file.
- `buttondown.md` — `build-publish` writes (the email body, byte-shaped like a raw Buttondown body: leading editor-mode comment, `---`-fenced parts intro → `## Currently` → cover → sections (with inline membership-block Liquid expansions + spliced-in featured sections at their declared `position`) → outro → haiku close → email-only tinylytics pixel; no block markers; the artifact `pipeline/content/content.py publish --issue N` pushes to Buttondown as a draft). The membership Liquid is audience-aware: a `cta:N` block shows the CTA only to non-premium subscribers (with Stripe upsell buttons for `regular`; subscribe form for anonymous); a `thanks:N` block shows the thanks only to `premium` members. `buttondown.html` alongside it is a Liquid-stripped preview rendered as a regular subscriber would see it.
- `cover.jpg`, `cover-large.jpg`, `journal/*.jpg`, `body-{N}.mp3`, `weekly-thing-{N}.mp3` — written by the iOS Shortcuts and `pipeline/audio/`; agents can't touch them (the `tools/s3.py` extension allowlist is text-only). `eddy-edits.md` — Eddy, rarely, for a preserved revision proposal.

**Issue-markdown shape (mirrors the iOS-Shortcut bodies in `data/buttondown/bodies/`).** Top-level blocks are `---`-fenced, in order: intro prose → `## Currently` (dropped when empty) → cover (`![](.../cover.jpg)` + caption + `Month D, YYYY  \nLocation`) → `## Notable` → optional featured section(s) → `## Journal` → optional featured section(s) → `## Briefly` → optional featured section(s) → outro prose (when present) → `A haiku to leave you with…` + the haiku + the closing "discuss on Reddit" line + `👨‍💻`. Featured sections are H2-level (`## {heading}`) and slot in after the named parent section based on the promoted row's `promoted_position`. The list loops have strict formatting:
- **Notable** — leads with `_You can discuss any of these links at the [Weekly Thing {N} tag in r/WeeklyThing](…flair_name%3A%22Weekly%20Thing%20{N}%22)._` (carries the issue number — generated, not templated), one blank line, then items. Each item is `### [Title](url)` then `\n\n{commentary}`. **Two blank lines between items.**
- **Briefly** — each item is `{commentary} → **[Title](url)**` (commentary first, then the `→` arrow, then the bolded link; no headings). **One blank line between items.**
- **Journal** — a status update (no title) is `[Weekday @ H:MM AM/PM](url)\n\n{content}` (e.g. `[Sunday @ 4:16 PM](url)`); a *titled* post is "elevated": `### [Title](url)  \n{Weekday @ H:MM AM/PM}\n\n{content}` (H3 link with a markdown hard break, label plain on the next line). The label is weekday-name + 12-hour clock — no date — because every entry sits inside the same seven-day issue window, so the weekday already identifies it. Photos already live inside `content_md` (rehosted, emitted as native `<img>` tags), each on its own paragraph. **Two blank lines between entries.**
- **Haiku** — `**line one  \nline two  \nline three**` (bold-wrapped, markdown hard break between lines).

Row-to-markdown renderers live in `tools/issue_items_render.py` (`render_notable` / `render_brief` / `render_journal`, `render_featured_section`, `reddit_tag_line`, plus `_with_markers` variants for create-final's inline cta/thanks splicing); `_base.format_haiku` is shared with `build-publish`. Final-document assembly is `tools/issue_assembly.py` (`assemble_final` for the block-markered final.md, `assemble_publish` for the strip-and-substitute buttondown.md). The membership CTA, the editor-mode comment, and the email tracking pixel are *not* in `draft.md` — they only appear in `buttondown.md` (the CTA composed by `compose-cta` and wrapped in the membership-block Liquid scaffold by `build-publish`; the editor-mode comment + tinylytics pixel added by `build-publish`). `update-draft` rebuilds `draft.md` from `templates/draft_starter.md` every run, so the layout always matches the template.

`tools/draft.py` (`section_status`) parses the draft for section item counts + asset presence + what's still missing for ship; `draft__section_status` is the agent tool over it. `tools/context.py` builds the per-persona dynamic-context blocks (`build_eddy_context` / `build_linky_context` / `build_patty_context` / `build_marky_context`) — runtime-computed facts injected into job prompts so the model doesn't recompute date math, word counts, queue depths, goal progress.

`tools/interaction.py` is the reaction primitive (`await_choice` / `await_approval`) — post options, wait for Jamie's reaction (filtered on `DISCORD_OWNER_USER_ID`), return the pick. Jobs hold their asset lock for the duration of the interaction.

## Conventions worth remembering

- **Pinboard tag convention is strict: only `_brief`.** Among the bookmarks `update-draft` projects into the issue: untagged-in-window = Notable; `_brief` = Briefly. The old `_featured` tag/section is retired. `client.capture_blurb` writes a blurb verbatim + tags `_brief` + clears `toread` atomically (still available for direct use). **`client.issue_window_candidates` only returns *public, read* bookmarks** — anything still flagged `toread` (Jamie hasn't worked through it) or marked private (`shared=no`) is excluded, so the draft never picks up an unreviewed link. (Linky's `pinboard-scan` still *sees* the `toread` pile — that's its job; the exclusion is only in the draft projection.) Linky's reply-to-card flow is the soft sibling: Jamie's Discord reply seeds the bookmark's description without tagging `_brief` or clearing `toread`, so he can finish curation in Pinboard.
- **`pinboard__popular_unseen` filters avoid-domains.** Before the `pinboard_popular_seen` dedup it drops items whose host is on `tools/avoid_domains.py`'s exclusion set (own domains, Buttondown, image CDNs, URL shorteners, social, Wikipedia, YouTube/Spotify/Vimeo, generic CDNs) — so Linky never even considers a "popular on Pinboard" Wikipedia article or t.co redirect as a possible Notable. That list is a hand-maintained *copy* of `pipeline/content/domain_exclusions.py` (`pipeline/` isn't importable from here); keep them loosely in sync — an exact match isn't required since this only gates Linky's scan, nothing it touches ships.
- **micro.blog: pull everything in window, no tag filtering** (`tools/microblog.py`). Uses the Micropub source query (`GET {MICROBLOG_MICROPUB_URL}?q=source`, `Authorization: Bearer MICROBLOG_API_KEY`) — returns posts as mf2-JSON with `properties.content` carrying the **native markdown Jamie wrote** (a string; micro.blog embeds photo uploads as `<img src=…>` tags inside it). `MICROBLOG_API_KEY` is **required** — no fallback; if micro.blog is down, `journal.fill` leaves a placeholder line (Eddy's review flags it). **`create-final` no longer cuts Journal entries** — Eddy is ordering-only. If an issue feels too long, handle upstream (delete from micro.blog before `update-draft` runs).
- **Journal images are rehosted and emitted as native `<img>` tags** (`tools/journal_images.py`, called from `_gather_fills`). Each `<img>` / `![]()` on an upload host (`MICROBLOG_IMAGE_HOSTS`, default `www.thingelstad.com` `/uploads/`, `micro.thingelstad.com`, `cdn.uploads.micro.blog`, `uploads.micro.blog`) is downloaded, resized to `MICROBLOG_IMAGE_MAX_DIM` (default 600px long side, downscale-only; JPEG q80 / PNG optimized), and copied into `weekly-thing/{N}/journal/<name>` (reusing the micro.blog hash basename), then the reference is rewritten to the local URL. Every output image — whether the source was a markdown `![]()` or a `<img>` tag — is emitted as a native `<img src="…" alt="…" />` with width/height/etc. preserved, each on its own paragraph (so adjacent gallery `<img><img>` don't run together). Going forward we use `<img>` for images everywhere we generate them (cover + journal), because it gives the alt attribute an explicit slot and leaves room for `width`/`height`/`loading`/`class` later; Eleventy and Buttondown both pass raw HTML through. After rehost/normalize, rehosted images with empty alts are filled via `tools.alt_text` (vision LLM, cached forever per content-addressed filename; per-run cap of 15 calls). A HEAD-skip on rehost + the alt cache make daily re-runs cheap; a per-image failure leaves that one's original URL alone, and an alt failure leaves the alt empty (the hygiene review picks it up). Binary writes go through `s3.write_journal_image` (image-only allowlist, `journal/` sub-prefix; *not* an agent tool). `update-draft` runs `_gather_fills` (the blocking source pulls + image rehosting + the HTML-preview write) via `asyncio.to_thread`.
- **Browser-viewable HTML previews.** `update-draft` / `create-final` / `build-publish` write a `.html` twin alongside their `.md` (`draft.html` / `final.html` / `buttondown.html` in the workspace) via `tools.render` (Python-Markdown → a self-contained styled page; the CSS mirrors `content/buttondown/newsletter/buttondown-email.css`; `draft`/`final` strip the block markers and get a "DRAFT…/FINAL…" banner, `publish` renders clean). They're uploaded with `Cache-Control: no-cache` and a CloudFront invalidation of that path (`tools.cdn.invalidate`, distribution `WEEKLY_THING_CDN_DISTRIBUTION_ID`, default the prod one; set to empty to disable) — so Jamie always sees the latest at `https://files.thingelstad.com/weekly-thing/{N}/draft.html` etc. The URL is surfaced in the job's result and (for `update-draft`, Tue–Fri) appended to Eddy's review card. Best-effort: a render/invalidation hiccup never fails the job. The pages carry a `<meta name="robots" content="noindex,nofollow">` and `files.thingelstad.com/robots.txt` is already `Disallow: /`, so the in-progress drafts aren't indexed (they *are* publicly reachable by URL — same as `draft.md` always has been on that public prefix).
- **The draft.html editorial-review drawer.** When `update-draft` produces a review (`prompts/eddy/draft-review.md` — a solid pass: anchored, concrete *suggestions only*, never a rewrite — run via `_draft_review` on sonnet, every run), `render.render_and_upload_html(..., review_md=…)` injects a fixed "Show review" button and a slide-in drawer on the right holding the rendered review. The drawer is **hidden by default** (`transform: translateX(100%)`, revealed by `body.rv-open` via a tiny inline `<script>` toggle) — so the same shareable URL reads as the clean draft until someone flips it on. When it's open and the viewport is wide enough (`min-width: 880px`), `body.rv-open` also shifts the draft column left (extra right margin reserved for the panel) so the review and the content sit side by side instead of the panel covering the text; on narrower windows the panel overlays. No review (empty draft → Eddy `PASS`, or no team) → no button, no drawer. The review prompt walks Intro → Notable → Journal → Briefly → Whole issue → **Hygiene** — that last pass is a deliverability-and-reader-skepticism lens (anchor text quality, anchor/domain mismatch, hype headings, spam-filter signal in voice, alt-text quality, subject-shaping section headings) folded into the same single Sonnet call, no separate scanner. The Discord `#editorial` card (`update-review.md`, Tue–Fri) is separate and unchanged — different audience, different cadence.
- **Off-loop source pulls.** Jobs that hit external APIs from inside the bot's asyncio loop wrap the blocking calls in `asyncio.to_thread` so a slow Pinboard/Buttondown/Tinylytics/RSS/S3 round trip doesn't stall the gateway: `update-draft` (`_gather_fills` — Pinboard + micro.blog + image rehost + HTML render), `pinboard-scan` (`build_linky_context`), `daily-metrics` (campaign polls + `subscriber_growth` + `tinylytics.summary` + `build_marky_context`), `promotion-prep` (RSS lookup + `buttondown.md` read + `build_marky_context`). Agent-loop calls (`bot.core`) are already thread-wrapped in `agent_loop.py`.
- **Marky operates on the last *published* issue**, not the in-flight one — the RSS feed (`tools/rss.py`) is her trigger; she reads `buttondown.md`.
- **`update-draft` is a pure projection** — never an additive merge. Real authoring lives upstream. Re-running it gives the same output modulo upstream changes.
- **Job prompts get a `## Today` dynamic-context block.** Read it; don't recompute date math / counts / queue depth / goal progress.
- Prompt name → file: `team` → `prompts/shared/team.md`; `<persona>` → `prompts/<persona>/prompt.md`; `<persona>-<file>` → `prompts/<persona>/<file>.md` (e.g. `eddy-update-review`, `linky-research-card`, `marky-promotion-prep`, `patty-compose-cta`). Edits need a bot restart (prompts are cached at first read).
- New schema → `db/schema.sql` (idempotent — re-runs every boot; the `goals` seed uses `WHERE NOT EXISTS`); new *columns* on existing tables → `db._COLUMN_MIGRATIONS`.
- Tests: `venv/bin/python -m unittest discover -s apps/workshop_bot/tests -t .` — run from the repo root; discord/anthropic/httpx stubbed (`tests/_stubs.py`); S3/Pinboard/micro.blog/Anthropic stubbed per-test. 650+ tests. Don't load `.env` for routine unit tests unless you specifically need runtime-config coverage.
