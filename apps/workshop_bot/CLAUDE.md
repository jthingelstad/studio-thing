# workshop_bot — project memory

> Author-only Discord bot for *The Weekly Thing*. Helps Jamie assemble each week's issue and ships supporting bits (link curation, membership CTA, promotion drafts). One Python process, several `discord.py` clients (one per persona, own token/avatar), plus an APScheduler instance — all in the same asyncio loop.

Full design rationale: [`notes/design/workshop-content-loop-design-brief.md`](../../notes/design/workshop-content-loop-design-brief.md). The user-facing overview is [`README.md`](README.md). This file is the "what to keep in mind when working here" memory.

**North star — [`docs/publishing-process.md`](../../docs/publishing-process.md):** the canonical model of how an issue is made + shipped. An issue moves through three per-issue **phases** — **Build → Publish → Share** — running *alongside* two standing **programs** (Patty's Membership drive, Marky's Campaigns) that touch a phase only at defined points. Each phase has a persistent card (Build/Publish in `#editorial`, Share in `#promotion`). The active issue carries `issue_windows.phase` ('build' | 'publish'); `mark built` (`/eddy issue built`) is the gate that opens Publish. Build never surfaces Publish concerns (subject/description/CTA) — that phase discipline is the fix for the old "Eddy nags about the subject mid-week" friction.

**Two operator surfaces, one model.** The **phase cards are the front door** — every routine action (refresh draft, reorder, haiku, subject, edit atom, ship a channel, draft promo) is a button on the relevant card. The **legacy slash commands stay as escape hatches** — `/eddy issue reorder`, `/eddy issue haiku`, `/eddy issue subject`, `/eddy issue publish [destination]`, `/eddy edit <asset>`, etc. still call the same compose-* / publish-* jobs directly, for repair scenarios (re-run one compose, ship one channel manually, debug a stuck atom). Both surfaces drive the same job runtime; the cards are the well-paved happy path. Don't add new "do a thing" commands as raw slashes — wire them through a card button so the operator's mental model stays card-first.

## Architecture: jobs are the spine

Every workshop_bot action — pulling content into the draft, reordering for the final, composing CTAs, drafting promotion copy — is a **job**: deterministic Python in `apps/workshop_bot/jobs/`, fired by a per-persona slash tree and/or by cron. Some jobs make small encapsulated LLM calls (Eddy's draft review, the compose-* jobs, Marky's reports); most are pure Python. The job is the unit of scheduling and on-demand execution.

Each persona hosts its own slash tree on its own Discord bot (no single `/workshop` umbrella anymore):

- `/eddy issue {start,update,status,build,echoes,built,reopen,reorder,haiku,subject,publish,put-to-bed,reset}` · `/eddy edit <asset>` · `/eddy currently {list,edit,set,clear,reorder,add-type,retire-type}` · `/eddy status` · `/eddy review <text>` · `/eddy archive <issue>` · `/eddy followup {list,add,cancel}`
- `/linky scan` · `/linky research <url>` · `/linky pile` · `/linky stats` · `/linky followup {list,add,cancel}`
- `/marky prep` · `/marky metrics` · `/marky engagement` · `/marky referrers` · `/marky campaign {add,edit,report,copy,sunset}` · `/marky followup {list,add,cancel}`
- `/patty cta` · `/patty goal {set,done}` · `/patty progress` · `/patty nonprofit` · `/patty supporters` · `/patty followup {list,add,cancel}`

The internal job names used below (and by the scheduler, the `job_locks` table, and `agent_runs`) are the dash-cased ones — `update-draft`, `reorder`, `compose-thesis`, `compose-haiku`/`-meta`/`-cta`, `render-audio`, `publish-{audio,buttondown,website}`, `reset-issue`, `edit-asset`, `pinboard-scan`, `promotion-prep`, `daily-metrics`, `follow-up-sweep`, etc. Slash aliases map to those: `/eddy issue update` → `update-draft`, `/eddy issue reorder` → `reorder` (DB row reorder only), `/eddy issue subject` → `compose-meta`, `/eddy issue publish [destination]` → `publish-*`, `/eddy issue reset` → `reset-issue`, `/eddy edit` → `edit-asset`, `/linky scan` → `pinboard-scan`, `/marky metrics` → `daily-metrics`, `/patty goal done` → `goal-achieved`, and so on. (`compose-thesis` fires automatically inside `mark-built`, no slash.)

The reader-facing Thingy bot — the public Q&A surface in `#ask-thingy` plus the operator-side `thingy-watch` conversation mirror + `/thingy {recent,show,sync}` commands — lives in a separate process at [`../thingy_bridge/`](../thingy_bridge/). See that app's CLAUDE.md for its architecture.

`jobs/_base.py` is the runtime: `JobContext` (carries `deps` — corpus/registry/team — and a `trigger` label; has `channel()` / `post()` helpers), `JobResult` (`{ok, message, data}` — `message` is acked to the invoker), single-asset locking (`job_lock([assets], name)` context manager backed by the `job_locks` SQLite table — two jobs that write the same file can't overlap; a lock held by a dead pid is stolen since a restart gets a new pid), and the draft-block helpers (`replace_block` / `get_block` for the `<!-- block:NAME -->` … `<!-- /block:NAME -->` markers in `draft.md`).

### The jobs

| Job | Trigger | What it does |
|---|---|---|
| `start-issue <n> <pub-date> <days>` (`/eddy issue start`) | manual | Records the issue window in `workshop.db`, refreshes the `weekly-thing/workshop.json` pointer Shortcuts read (issue number, dates, predictable workspace URLs), seeds `draft.md` from `templates/draft_starter.md`, auto-fires `update-draft`. The only job that takes the issue number explicitly. |
| `update-draft` | daily 17:00 CT + after `start-issue` + manual | Pure projection: rebuilds `draft.md` from `templates/draft_starter.md` every run (so the layout always tracks the template) and fills each block. Notable/Briefly/Journal come from `issue_items` rows — `issue_items_sync.sync_all` first refreshes rows from Pinboard + micro.blog (UPSERT on stable source ids; non-promoted rows whose upstream item disappeared are pruned), then `issue_items_render` projects rows back to section markdown. Atoms stay file-backed under `s3://files.thingelstad.com/weekly-thing/{N}/atoms/` — `intro.md`, `outro.md`, `haiku.md`, `cover.json`, `metadata.json`, `thesis.md`, `cta-1.md`, `cta-2.md`, `thanks-1.md`. Then **the three sibling artifacts fire** via `tools/renderers.render_all_for_issue` — `archive.md` + `links.json`, `buttondown.md`, `transcript/*.txt` — so the daily output always reflects current DB + atom state (day 1 of a new issue produces placeholder-filled artifacts, filled in as content arrives). Each renderer is a pure function over the structured input; they share no intermediate text body. After the renders, a **single solid editorial review** of the draft (`prompts/eddy/draft-review.md`, **Opus**, `_DRAFT_REVIEW_DEFAULT_MODEL`) gets embedded in `draft.html` behind a "Show review" toggle — hidden by default, so the shareable link reads clean until someone flips it on (`_draft_review` → `render.render_and_upload_html(review_md=…)`); each target-anchored bullet in the review is captured to `editorial_comments` with a stable handle (`E349-N1`, `E349-X3`, etc.) and rendered as a copy-button badge in the drawer so Jamie can paste the handle into Discord to continue the conversation (`@eddy tell me about E349-N1` resolves via the `editorial__get_comment` tool); writes a `draft_digests` row. **The old second review** — a freeform weekday-scaled `#editorial` card on `update-review.md` — was **retired** (different prompt/model, unstored, it contradicted the drawer and re-surfaced already-closed comments); the open-comment count now shows on the Build card instead. The Opus review is a **Build-phase** concern: `update-draft` runs it only when `phase='build'` (during Publish the content is frozen, so it's skipped). **No-op short-circuit:** when the reprojected draft is byte-identical to the last run (`source_hash` matches the prior `draft_digests` row), the Opus review + HTML re-render are skipped (the phase card is still refreshed) — a no-op cron tick no longer floods or burns an LLM call. There is no longer a "final.md exists" lock — the issue can be re-published any time, all subcommands are idempotent. **Instead of the old per-tick `📋` status card, `update-draft` refreshes the phase-appropriate card** (`_refresh_phase_card` → Build card during build, Publish card during publish). |
| `issue-status` | manual | Read-only section + asset completeness report (`render_status_card`). The same checklist now lives, always-current, on the Build/Publish cards. |
| **phase cards** (`build-card` / `publish-card` / `share-card`) | start-issue posts Build; `mark built` posts Publish; put-to-bed posts Share; each `update-draft` tick refreshes the phase-appropriate card | **One persistent, self-editing card per phase** (`jobs/{build,publish,share}_card.py`; shared lifecycle in `jobs/_cards.py`; views in `personas/views/{build,publish,share}_card_view.py`; persistent View pattern + per-phase card handles in the `issue_cards` table). State is a pure function of DB + S3 — no wizard-state table. **Build** (`#editorial`, Eddy): the **authored** content anatomy in reading order (Intro·Currently·Cover·Notable·Journal·Briefly·Outro·**Echoes**) + review + reorder; buttons Refresh/Reorder/Echoes/Edit/**Mark built** (gated on content completeness). Haiku is NOT here — Eddy writes the haiku, not Jamie, so it's a shipping concern. **Publish** (`#editorial`, Eddy): **Eddy's thesis** at the top (written automatically at mark-built — the editorial framing that anchors every Publish job) + shared envelope (subject/description/haiku/CTA, all auto-requestable or generated) + the per-channel matrix (Email/Website/Podcast); 🚀 buttons **disabled until their gate passes**. **Share** (`#promotion`, Marky): the last-published issue's syndication launchpad (Draft promo / Refresh metrics) + campaign context. All three button Views are `timeout=None`, registered via `EddyBot`/`MarkyBot.persistent_views` → `bot.add_view` so clicks route across restarts; each button launches the existing job in the background then re-renders the card. |
| `reorder` (`/eddy issue reorder`) | manual | Eddy's Notable/Brief reorder pass. **The LLM specifies the order; code mutates rows.** Reads `issue_items` rows (Notable / Brief — Journal is time-ordered by sync, never reordered), surfaces them to Eddy with stable synthetic ids (`n1`/`b2`), asks for an ordering-only JSON proposal (`{notable_order, brief_order}`), validates strictly (each `*_order` is a strict permutation of the section's row ids), then applies: `reorder(issue, section, ordered_row_ids)` per section. Promotions (Featured) come from the micro.blog `Featured` category at sync time, not Eddy. Writes a **side-by-side `final-proposal.html`** preview so Jamie can read the reorder visually before reacting. Posts to `#editorial`: per-section was/now reorder map + 📄 link to the side-by-side view; Jamie ✅/❌/🔄. On ✅: applies the row mutations + fires `renderers.render_all_for_issue` so the daily artifacts immediately reflect the new ordering. On ❌: leaves rows as-is. Reorder does **not** write `thesis.md` — that's `compose-thesis`'s job, fired at `mark-built`. |
| `compose-thesis` | auto-fired inside `mark-built` | One-shot LLM call. Reads the just-frozen issue (`draft.md`, intro + sections in their post-reorder sequence + outro) and writes a 1–3 sentence editorial framing to `atoms/thesis.md`. Downstream prompts — `compose-meta:subject` / `compose-meta:description` / `compose-haiku` / `compose-cta` — read it as a `## Thesis` prefix in their LLM messages so the four shipping jobs land on the same editorial anchor. Best-effort: failure logs but doesn't wedge the Build → Publish transition (downstream prompts degrade gracefully on missing `thesis.md`). Jamie can refine via `/eddy edit thesis`. `jobs/compose_thesis.py`; prompt `prompts/eddy/compose-thesis.md`. |
| `compose-haiku` | manual (Publish-card button) | Eddy returns JSON haiku options → `#editorial`; Jamie picks → `haiku.md`. The pick prompt also links a `haiku-options.html` page (option cards with a copy button per haiku, mono font preserving the line breaks). Required for ship — Eddy writes the haiku (not Jamie), so this is a Publish-phase concern alongside subject/description/CTA. |
| `compose-meta` | manual | Two LLM steps in `#editorial`. **Subject:** Eddy returns **5 `WT<N> — <theme>` options** (`prompts/eddy/compose-subject.md` — Jamie's prompt, verbatim) → Jamie reacts 1️⃣–5️⃣ (🔄 = refresh). The pick prompt also links a `subject-options.html` page (5 cards with a copy button each, serif body) so Jamie can read the full lines without Discord wrapping them. **Description:** Eddy generates a single comma-separated topic line (`prompts/eddy/compose-description.md` — also Jamie's prompt, verbatim) — *no picker*, the prompt is deterministic enough (length window, ordering rules, style guardrails) for one-shot. The success post surfaces both so Jamie can spot a bad description and edit in Buttondown or re-run. Writes `metadata.json` (subject/description generated; image/slug/number/publish_date deterministic). Required for ship. |
| `compose-cta` (`/patty cta`) | manual | Patty always composes three atom files per issue: `cta-1.md`, `cta-2.md`, `thanks-1.md`. Generates 1–2 framings per slot (kind-aware: supporter CTA in Thingy's voice for `cta:N`, sincere thank-you for `thanks:N`), Jamie picks per slot via the standard refresh-loop UX → writes the atom file with `kind:` YAML frontmatter (`supporter` or `thanks`). Each per-slot pick prompt also links a `{kind}-{slot_n}-options.html` cards page. **Placement is hardcoded in `render_email`** via the `CTA_SLOT_POSITIONS` constant — `cta:1` splices after Notable, `cta:2` after Journal, `thanks:1` after Briefly. Slots whose atom file is missing/empty produce no marker in the email. Patty doesn't decide count or placement; Eddy doesn't either. Already-filled slots are skipped on re-run; delete a file to re-roll (or use `/eddy edit cta-N`). Thesis (`thesis.md`) injected into both prompts when present. |
| `render-audio` (`/eddy issue publish audio`) | manual | Wraps `pipeline/audio/audio.build_issue`: TTSes each per-block transcript via OpenAI, concats with bumpers + ID3 + loudnorm, uploads MP3 to `s3://files.thingelstad.com/weekly-thing/{N}/weekly-thing-{N}.mp3`, updates the local `data/audio/manifest.json` with audio_url + duration + byte_size + script hash. Idempotent: `build_issue`'s freshness gates skip the render when the transcript hash matches the manifest and the S3 MP3 is intact. Slowest step in the ship (~30–60s for TTS + S3 upload). Requires `OPENAI_API_KEY` + AWS creds for S3. `jobs/render_audio.py`. |
| `compose-echoes` | auto-fired inside `mark-built` (Build → Publish transition) | Generates **Echoes** — Thingy's archive note that closes every issue. Reads the built body, last 6 echoes notes (anti-repetition), top archive snippets via Thingy's `/retrieve` (semantic), and anniversary candidates (1/5/8 years back via `apps/site/_data/emails.json` + per-issue body previews from `data/issues/{N}/archive.md`). Opus picks mode 1 (thematic resonance, 1-3 issues) or mode 2 (anniversary echo), writes 2-5 sentences in Thingy's voice (third-person Jamie), and references issues as `[WT###](https://weekly.thingelstad.com/archive/N/)` markdown links. Post-processor linkifies any bare references. **Fails loud** on retrieval failure — quality bar requires real semantic retrieval. Mandatory — no SKIP. Writes `echoes.md` to S3 + local `data/issues/{N}/atoms/`. The renderers splice `## Echoes` between haiku and the Reddit-discuss line. `jobs/compose_echoes.py`; prompt `prompts/eddy/compose-echoes.md`. |
| `publish-{audio,buttondown,website}` (`/eddy issue publish [destination]`) | manual | Destination-aware ship. Each subcommand ships to one destination and is independently idempotent. **`publish audio`** = render-audio (TTS + S3 upload + manifest update). **`publish buttondown`** = pre-flight required-atom check (`haiku.md`, `metadata.json`, `intro.md`, `cover.jpg`), POST or PATCH the daily-rendered `buttondown.md` to Buttondown's API, capture `buttondown_id` + `absolute_url` into `metadata.json`, then re-render the archive so its front matter carries the new URL. **`publish website`** = single atomic GitHub commit of `data/issues/{N}/{archive.md, metadata.json, links.json, transcript/*.txt, echoes.md}` + `data/audio/manifest.json`. **`publish all`** = audio → buttondown → website (in that order — audio first so archive carries audio fields, website last so the commit has the absolute_url). Every subcommand assumes `update-draft` ran recently — the daily renderers write the artifacts each subcommand reads. `jobs/publish.py`. Requires `GITHUB_PAT_TOKEN` env var (fine-grained PAT, Contents: write on `jthingelstad/weekly.thingelstad.com`). |
| `reset-issue {final\|publish}` (`/eddy issue reset`) | manual | Drops previous-step artifacts so the in-flight issue can be re-published. `reset reorder` (internally `step="final"`) clears `issue_items` promotions and deletes `thesis.md` so the next `/eddy issue reorder` proposes against the current row state. `reset publish` deletes `buttondown.md` from S3 so the next `update-draft` tick re-renders. Deliberately preserved: `issue_items` rows themselves, `cta-N.md` / `thanks-N.md` files, and `metadata.json`'s `buttondown_id` (so the next `publish buttondown` PATCHes the same draft rather than orphaning it with a fresh POST). Idempotent. |
| `edit-asset <asset>` (`/eddy edit`) | manual | Pops a Discord modal pre-filled with the atom's current S3 contents; submit writes back. Asset is one of `intro`/`outro`/`haiku`/`cover`/`cta-1`/`cta-2`/`thanks-1`. For atoms that flow into `draft.md` (`intro`/`outro`/`haiku`/`cover`), the job also re-fires `update-draft` so the four artifact previews refresh. Discord modals cap at 4000 chars per `TextInput`, which fits every atom file; bodies over the cap refuse with a "use the S3 console" hint. Deliberately *not* editable here: `draft.md` / `archive.md` / `buttondown.md` / `transcript/*.txt` (all generated; regenerate from atoms via `/eddy issue update`), `metadata.json` (structured; subject/description come from `compose-meta`). |
| `currently {list,edit,set,clear,reorder,add-type,retire-type}` (`/eddy currently`) | manual + Mon/Wed evening cron via `follow-up-sweep` | DB-backed `## Currently` editor. `list` shows the in-flight issue's filled entries + unfilled active types + stale-pick hint. `edit <type>` pops a per-type modal pre-filled with the current DB value (good for values carrying markdown links). `set` is the non-modal quick path; `clear` deletes one and renumbers; `reorder` takes a comma-separated permutation of filled labels. `add-type`/`retire-type` manage the canonical pool — adding "Printing" is just `/eddy currently add-type Printing`, no code change. Mutating ops write to the DB and stop there — the next scheduled (daily 17:00 CT) or manual `/eddy issue update` projects the change into `draft.md`. Eddy also handles this conversationally in `#editorial` via the `currently__*` agent tools (`set` / `clear` / `reorder` / `add_type` / `suggest_stale` / `list_types` / `list_entries`); Mon 17:00 CT + Wed 17:00 CT follow-ups (scheduled by `start-issue`) seed the week's conversation, asking Jamie about a stale type. The legacy iOS-Drafts → Shortcut → `currently.json` path is retired. |
| `feedbin-ingest` | hourly at :35; manual | Mirrors Jamie's Feedbin starred-items feed into Pinboard as `toread=yes shared=yes` bookmarks. Reads `FEEDBIN_STARRED_FEED_URL`; dedups via `feedbin_starred_seen` (GUID-keyed); calls `pinboard.posts_add` per unseen item. PASSes silently when no new stars; posts a brief summary to `#chatter` (status, not actionable — the bookmarks themselves surface as `#research` toread cards on the next `pinboard-scan`) only when bookmarks were actually filed. `jobs/feedbin_ingest.py`. |
| `pinboard-scan` | every 3h 07:00–22:00 CT (07/10/13/16/19/22), year-round; manual | Linky's per-link research pass, posting cards into **two channels by source**: toread items (Jamie's own Pinboard adds + the Feedbin starred mirror) → `#research` (env `DISCORD_CHANNEL_RESEARCH`); discovery items (Pinboard popular + future feeds) → `#discovery` (env `DISCORD_CHANNEL_DISCOVERY`). One **toread** lane (Jamie's own public-`toread` bookmarks not yet in `pinboard_research_done`, via `pinboard.toread_public_unresearched`) plus N **discovery feeds** declared in the `DISCOVERY_FEEDS` registry in `tools/feeds/feed_registry.py` — each entry is a `FeedSpec` carrying name, label, pin-label, fetch callable, per-scan cap, feed limit, and primary priority. Today's discovery set is **Pinboard popular only**; the registry remains generic so future feeds can be added intentionally, while retired non-Pinboard feed rows are removed by the DB migration. Cross-source mechanics still exist for the toread lane and any future feed: a **cross-day uplift** (URL was seen on a different lane/feed days ago → `## Cross-source uplift` block carries the prior sightings + original verdict; Linky reconsiders, capped at `_UPLIFT_PER_SCAN_CAP=5` per scan; sightings tracked in the `popular_seen_sightings` table). Uplift cards inherit their primary source's routing — a `popular`-source uplift lands in `#discovery`. To add a feed: write `tools/feeds/<name>.py` with `top(limit=N)` (or equivalent) returning `[{url, title, [discussion_url], [score], [comment_count], [submitter], [tags]}]`, add one `FeedSpec` line, append the name to `db.RESEARCH_SOURCES`. The job (not the LLM) iterates candidates and calls Linky's `core` once per link with `prompts/linky/research-card.md` (source-agnostic — switches on `toread` vs. discovery via the `## The link` block's `Source` field); each per-link call returns either a Discord card or one of two signals — `SKIP: <reason>` (discovery sources only — judged not interesting; URL is marked seen and never resurfaces from the same feed, may still uplift if a *different* lane/feed surfaces it later) or `FETCH_FAILED: <reason>` (any source; URL is NOT marked, so it can retry next scan). Cards no longer include read-length/source footers or "also on" lines. Each card posts as its own message in the routed channel; the `discord_message_id` is recorded in `linky_research_messages` so a Discord reply / reaction can route back to the right Pinboard bookmark via the row's `source`. URL normalisation for cross-source dedup is `tools/url_normalize.py`'s `dedup_key` (lowercases host, strips trailing slash, drops `utm_*`/`fbclid`/etc. — in-memory comparison only; stored URLs are unchanged). **Archive resonance** runs as a Python pre-step on every card: `thingy_retrieve.retrieve(query, k=3)` (semantic — Bedrock Cohere embed + rerank via Thingy's `/retrieve`), results rendered as a `## Archive resonance` block (≤180-char snippets per hit) inside the `## The link` block. Query strategy is `title + description` for toread, `title` for discovery sources. Fail-soft: a Lambda hiccup logs and omits the block — the hourly scan keeps moving. The prompt explicitly says resonance is informational, not a filter. The LLM still has `archive__retrieve` (semantic) and `archive__search` (lexical) as tools for follow-up phrasing if the obvious lookup misses. PASSes silently when all sources come back empty (common off-hour). |
| `promotion-prep` | auto-fired by `put-to-bed` (issue enters Share); manual (`/marky prep` or the Share card) | Marky drafts syndication content (LinkedIn + r/WeeklyThing megathread + per-link threads, 2–3 framings each) for the **last-published** issue (`db.get_latest_issue()`)'s `buttondown.md` → `#promotion`. Never auto-posts. |
| `daily-metrics` | daily 19:00 CT; manual | Polls active campaigns (Tinylytics `?ref=` traffic + Buttondown ref signups → a `campaign_metrics` row each run), checks subscriber growth + engagement; PASSes silently when nothing material moved, else Marky composes a terse report → `#promotion`. |
| `add-campaign <name> <ref> [signups] [traffic] [copy]` | manual | Inserts a row into the `campaigns` table. `ref` is stored verbatim (case-sensitive — Tinylytics matches the `?ref=` value exactly). `copy` (optional) is the actual promo text that ran in the placement. |
| `campaign-edit <name> [ref] [started_at] [ends_at] [signups] [traffic] [notes] [copy]` | manual | Updates an existing campaign's editable fields in place — only the ones you pass (`db.CAMPAIGN_EDITABLE`); the name is immutable and `status` flips via `campaign-sunset`. In `jobs/ops.py` → `db.update_campaign`. |

| `goal-achieved [notes]` | manual | Marks the active goal achieved (today); `notes` is appended to whatever was recorded at `set-goal` time. |
| `follow-up-sweep` | hourly (cron `23 * * * *`) | Fires due **follow-ups** — commitments an agent (or Jamie) made via `followup__schedule` / `/<persona> followup add`, stored in the `follow_ups` table. Two trigger kinds: time-based (`due_at` — any distance: tomorrow, next month, next year) and issue-based (`trigger_issue` — fires once the active in-flight issue reaches it). For each due one: runs the persona's agent loop with the note + a `## Today` context block, posts the agent's check-in to the persona's home channel, marks it fired. PASSes silently when nothing's due; an agent PASS marks it fired without posting; an agent/channel error leaves it open to retry; ≤8/sweep. `jobs/follow_up.py`. **This is the deliberate, targeted replacement for per-persona heartbeats** — it only does anything when a real commitment comes due. `jobs/follow_up.py` also backs each persona's `/<persona> followup {list,add,cancel}` slash subgroup and the `followup__schedule` / `followup__list` / `followup__cancel` agent tools. |

Plus one **top-level cross-cutting** subcommand on Eddy: **`/eddy status`** — a read-only DB-only ops snapshot (active issue window, active goal/campaigns, any held `job_locks`, the last few `agent_runs`). Distinct from `/eddy issue status` (the in-flight issue's *content* completeness) and `/marky campaign report` (campaign perf vs expected). Source: `jobs/status.py`; `set-goal`/`goal-achieved`/`campaign-sunset`/`campaign-copy`/`campaign-edit` are in `jobs/ops.py` (tiny, no-LLM).

Scheduler: `scheduler/jobs.py` declares the cron `JobSpec`s; `scheduler/handlers.py` has `content_job` (the bridge cron→jobs, wired as `functools.partial(content_job, job="<name>")`). **Sharing is phase-driven, not polled** — there's no RSS-poll job; `put-to-bed` (issue enters Share) posts the Share card and auto-fires `promotion-prep`. There are **no per-persona heartbeats** — everything an agent does on a cadence is a job. The closest thing to "an agent acting on its own" is `follow-up-sweep` firing a commitment the agent itself scheduled.

Slash dispatch (`personas/commands/`): one register fn per persona — `register_{eddy,linky,marky,patty}_commands(bot)` — each builds and adds a top-level group to that persona's `CommandTree`. Shared dispatch helpers live in `personas/commands/_shared.py` (`make_ack`, `make_run_and_ack`, `make_run_interactive`, `_ctx`). The fast jobs `defer` → run → ack with the result (the ack send is wrapped in try/except — a Discord interaction token only lasts ~15 min). The **interactive** jobs (`reorder`, `compose-haiku`/`-meta`/`-cta` — they post options to a channel and wait for Jamie's reaction, which can outlast the token) instead ack *immediately* ("started — react in #editorial / #supporters") and then run; the job posts its own outcome to the channel, so no second followup is needed. `/eddy status` is attached directly under the `/eddy` group rather than under `/eddy issue …` (it's a bot-health view, not tied to a single artifact). Each persona's `PersonaBot.__init__` calls its register fn and the base class syncs the tree on `on_ready`.

### The unified asset pattern

Every piece of issue content is a standalone file in the S3 workspace `s3://files.thingelstad.com/weekly-thing/{N}/`:

- `intro.md` (the opener prose) and `outro.md` (the closing prose) — both Jamie-authored, pushed via Shortcut; `update-draft` reads each verbatim into its block (`intro` / `outro`). The cover block's caption/date/location comes from `cover.json` — `{"caption":…,"location":…,"timestamp":…}` rendered as `caption\n\ntimestamp  \nlocation` with the derived `![](.../cover.jpg)` prepended (`jobs/_cover.py`); a legacy verbatim `cover.md` (caption + `date  \nlocation`) is still accepted. The optional "Currently" section is **DB-backed** — one row per type per issue in `currently_entries`, ordered by `position` (insertion-order by default; reorderable by Eddy/Jamie). `jobs/_currently.render` reads from the DB and emits `**Label:** value` lines, one blank line between. The canonical type pool lives in `currently_types` (seed: Listening, Watching, Reading, Playing, Installing, Dining, Cooking, Making, Drinking, Printing). The retired iOS-Drafts → Shortcut → `currently.json` path is no longer read.
- `haiku.md` — `compose-haiku` writes (rendered bold + hard-break by `_base.format_haiku` at render time); `metadata.json` — `compose-meta` writes (subject/description/image/slug/publish_date); `publish buttondown` writes back the `buttondown_id` field on the first successful POST so re-runs PATCH the same draft. `cta-N.md` / `thanks-N.md` — `compose-cta` writes (1–2 supporter CTAs + 0–1 thank-you, each with `kind: supporter` / `kind: thanks` YAML frontmatter; the audience-aware Liquid wrapping happens at render time inside `render_email`).
- `thesis.md` — `compose-thesis` writes at `mark-built` (the Build → Publish phase transition). One to three sentences in Eddy's voice naming the editorial intent for the now-frozen issue. Read by `compose-meta`, `compose-haiku`, and `compose-cta` as a `## Thesis` prefix in their LLM messages so the subject options, description, haiku, and CTA framings all anchor on the same narrative. Editable via `/eddy edit thesis`.
- `draft.md` — `update-draft` writes (a regenerable pure projection of all the above + upstream, shaped like a delivered issue).
- **There is no `final.md` artifact.** `/eddy issue reorder` mutates `issue_items.position` in the DB and writes `thesis.md` — the daily renderers consume the row state directly. The side-by-side `proposal.html` rendered by `create-final` is the editorial-review preview.
- `buttondown.md` — `tools/renderers.render_email_for_issue` writes (the email body, byte-shaped like a raw Buttondown body: leading editor-mode comment, `---`-fenced parts intro → `## Currently` → cover → sections (with inline membership-block Liquid expansions at hardcoded `CTA_SLOT_POSITIONS` + spliced-in featured sections from upstream `Featured` tags) → outro → haiku close → email-only tinylytics pixel; no block markers). The membership Liquid is audience-aware: a `cta:N` block shows the CTA only to non-premium subscribers (with Stripe upsell buttons for `regular`; subscribe form for anonymous); a `thanks:N` block shows the thanks only to `premium` members. There is no `buttondown.html` preview — `buttondown.md` is a ship-shaped artifact for Buttondown's API, not a review surface (the review surface is `draft.html`).
- `archive.md` — `tools/renderers.render_archive_for_issue` writes the website-shaped body (full YAML front matter + body, no editor-mode preamble, no Tinylytics pixel, no cta/thanks). The canonical website source — `publish website` commits this verbatim to `data/issues/{N}/archive.md` in the repo. `links.json` — same renderer writes the structured link extraction (`notable_links`, `briefly_links`, `domains`, `word_count`) for downstream JSON consumers. `transcript/NNN-{slug}.txt` — `tools/renderers.render_transcript_for_issue` writes one file per semantic block; the audio pipeline TTSes each as its own utterance.
- `cover.jpg`, `cover-large.jpg`, `journal/*.jpg`, `body-{N}.mp3`, `weekly-thing-{N}.mp3` — written by the iOS Shortcuts and `pipeline/audio/`; agents can't touch them (the `tools/s3.py` extension allowlist is text-only). `eddy-edits.md` — Eddy, rarely, for a preserved revision proposal.

**Issue-markdown shape (mirrors the iOS-Shortcut bodies in `data/buttondown/bodies/`).** Top-level blocks are `---`-fenced, in order: intro prose → `## Currently` (dropped when empty) → cover (`![](.../cover.jpg)` + caption + `Month D, YYYY  \nLocation`) → `## Notable` → optional featured section(s) → `## Journal` → optional featured section(s) → `## Briefly` → optional featured section(s) → outro prose (when present) → `A haiku to leave you with…` + the haiku + the closing "discuss on Reddit" line + `👨‍💻`. Featured sections are H2-level (`## {heading}`) and slot in after the named parent section based on the promoted row's `promoted_position`. The list loops have strict formatting:
- **Notable** — leads with `_You can discuss any of these links at the [Weekly Thing {N} tag in r/WeeklyThing](…flair_name%3A%22Weekly%20Thing%20{N}%22)._` (carries the issue number — generated, not templated), one blank line, then items. Each item is `### [Title](url)` then `\n\n{commentary}`. **Two blank lines between items.**
- **Briefly** — each item is `{commentary} → **[Title](url)**` (commentary first, then the `→` arrow, then the bolded link; no headings). **One blank line between items.**
- **Journal** — a status update (no title) is `[Weekday @ H:MM AM/PM](url)\n\n{content}` (e.g. `[Sunday @ 4:16 PM](url)`); a *titled* post is "elevated": `### [Title](url)  \n{Weekday @ H:MM AM/PM}\n\n{content}` (H3 link with a markdown hard break, label plain on the next line). The label is weekday-name + 12-hour clock — no date — because every entry sits inside the same seven-day issue window, so the weekday already identifies it. Photos already live inside `content_md` (rehosted, emitted as native `<img>` tags), each on its own paragraph. **Two blank lines between entries.**
- **Haiku** — `**line one  \nline two  \nline three**` (bold-wrapped, markdown hard break between lines).

Row-to-markdown renderers live in `tools/issue_items_render.py` (`render_notable` / `render_brief` / `render_journal`, `render_featured_section`, `reddit_tag_line`); `_base.format_haiku` is shared with `render_email`. The three pure format renderers live in `tools/renderers.py` (`render_archive`, `render_email_body`, `render_transcript_blocks`) — each consumes the same structured input (atoms + sections + features + metadata + cta_atoms) and emits its format directly. No shared intermediate text body, no marker-substitution machinery, no marker-stripping between formats. The membership CTA, the editor-mode comment, and the email tracking pixel are *not* in `draft.md` or `archive.md` — they only appear in `buttondown.md`. `update-draft` rebuilds `draft.md` from `templates/draft_starter.md` every run, then fires the three sibling renderers so all four artifacts stay current.

`tools/draft.py` (`section_status`) parses the draft for section item counts + asset presence + what's still missing for ship; `draft__section_status` is the agent tool over it. `tools/context.py` builds the per-persona dynamic-context blocks (`build_eddy_context` / `build_linky_context` / `build_patty_context` / `build_marky_context`) — runtime-computed facts injected into job prompts so the model doesn't recompute date math, word counts, queue depths, goal progress.

`tools/discord/interaction.py` is the picker primitive (`await_choice` / `await_approval`) — post options as **buttons** (numbered + Refresh, or Accept/Skip/Refresh), wait for Jamie's click (filtered on `DISCORD_OWNER_USER_ID`), return the pick. Same signatures + return contract as the old reaction version (callers + their mocks unchanged); a transient owner-filtered `_PickView` resolves an asyncio.Event. Jobs hold their asset lock for the duration of the interaction.

## Conventions worth remembering

- **Model tiers: Sonnet default, Opus for editorial, Haiku for bulk.** Every persona's `preferred_model` is **Sonnet** — that's the floor for general work (mentions, `#team` Q&A, composition, follow-ups, ad-hoc agent loops). **Opus** is reserved for editorial-grade judgment calls and is selected by explicit `model="opus"` overrides at the callsite: today that's `update-draft:html-review` (the canonical draft.html drawer review), `create-final` (reorder + thesis), and `review-text` (`/eddy review <text>` ad-hoc pass). **Haiku** is the override DOWN for high-volume bulk where SKIP is the majority outcome: today that's the discovery lane of `pinboard-scan` (Pinboard popular feed; future discovery feeds inherit). Toread lane stays on the Sonnet default — Jamie's own picks deserve real research quality. When a new job ships, ask "is this editorial judgment, general work, or bulk filtering?" and apply the corresponding tier — don't reach for Opus by default.
- **Channel routing: actionable in the home channel, status in `#chatter`.** Each persona's home channel (`#editorial` / `#research` / `#discovery` / `#promotion` / `#supporters`) carries things Jamie acts on — per-link cards, picks, drafts, reorder approvals, ship buttons. Status output that just says "I did X" with no follow-up needed goes to `#chatter` so the home channels don't get noisy. Examples on the chatter side: every persona's startup announcement; the pinboard-scan one-line summary (popular reviewed/posted/skipped, or feed-down alert); the feedbin-ingest "filed N new toread bookmark(s)" summary (the bookmarks themselves are actionable and surface as `#research` cards on the next scan — only the meta-status is chatter).
- **Pinboard cadence is enforced, not just logged.** `systems/pinboard/client._throttle(family)` sleeps the gap between calls when needed, per the documented per-endpoint cadence (`/posts/all` 5min, `/posts/recent` 60s, everything else 3s). Per-family `_CADENCE_SLEEP_CAP`: `standard` always sleeps (≤3s), `recent` sleeps up to 5s then warns, `all` never sleeps (would block the loop ~5min) — it warns instead because a sub-5min `/posts/all` is a structural bug (a redundant call to remove, not a wait to paper over).
- **Pinboard tag convention is strict: only `_brief`.** Among the bookmarks `update-draft` projects into the issue: untagged-in-window = Notable; `_brief` = Briefly. The old `_featured` tag/section is retired. `client.capture_blurb` writes a blurb verbatim + tags `_brief` + clears `toread` atomically (still available for direct use). **`client.issue_window_candidates` only returns *public, read* bookmarks** — anything still flagged `toread` (Jamie hasn't worked through it) or marked private (`shared=no`) is excluded, so the draft never picks up an unreviewed link. (Linky's `pinboard-scan` still *sees* the `toread` pile — that's its job; the exclusion is only in the draft projection.) Linky's reply-to-card flow is the soft sibling: Jamie's Discord reply seeds the bookmark's description without tagging `_brief` or clearing `toread`, so he can finish curation in Pinboard.
- **`pinboard__popular_unseen` filters avoid-domains.** Before the `pinboard_popular_seen` dedup it drops items whose host is on `tools/avoid_domains.py`'s exclusion set (own domains, Buttondown, image CDNs, URL shorteners, social, Wikipedia, YouTube/Spotify/Vimeo, generic CDNs) — so Linky never even considers a "popular on Pinboard" Wikipedia article or t.co redirect as a possible Notable. That list is a hand-maintained *copy* of `pipeline/content/domain_exclusions.py` (`pipeline/` isn't importable from here); keep them loosely in sync — an exact match isn't required since this only gates Linky's scan, nothing it touches ships.
- **micro.blog: pull everything in window, no tag filtering** (`tools/microblog.py`). Uses the Micropub source query (`GET {MICROBLOG_MICROPUB_URL}?q=source`, `Authorization: Bearer MICROBLOG_API_KEY`) — returns posts as mf2-JSON with `properties.content` carrying the **native markdown Jamie wrote** (a string; micro.blog embeds photo uploads as `<img src=…>` tags inside it). `MICROBLOG_API_KEY` is **required** — no fallback; if micro.blog is down, `journal.fill` leaves a placeholder line (Eddy's review flags it). **`create-final` no longer cuts Journal entries** — Eddy is ordering-only. If an issue feels too long, handle upstream (delete from micro.blog before `update-draft` runs).
- **Journal images are rehosted and emitted as native `<img>` tags** (`tools/journal_images.py`, called from `_gather_fills`). Each `<img>` / `![]()` on an upload host (`MICROBLOG_IMAGE_HOSTS`, default `www.thingelstad.com` `/uploads/`, `micro.thingelstad.com`, `cdn.uploads.micro.blog`, `uploads.micro.blog`) is downloaded, resized to `MICROBLOG_IMAGE_MAX_DIM` (default 600px long side, downscale-only; JPEG q80 / PNG optimized), and copied into `weekly-thing/{N}/journal/<name>` (reusing the micro.blog hash basename), then the reference is rewritten to the local URL. Every output image — whether the source was a markdown `![]()` or a `<img>` tag — is emitted as a native `<img src="…" alt="…" />` with width/height/etc. preserved, each on its own paragraph (so adjacent gallery `<img><img>` don't run together). Going forward we use `<img>` for images everywhere we generate them (cover + journal), because it gives the alt attribute an explicit slot and leaves room for `width`/`height`/`loading`/`class` later; Eleventy and Buttondown both pass raw HTML through. After rehost/normalize, rehosted images with empty alts are filled via `tools.alt_text` (vision LLM, cached forever per content-addressed filename; per-run cap of 15 calls). A HEAD-skip on rehost + the alt cache make daily re-runs cheap; a per-image failure leaves that one's original URL alone, and an alt failure leaves the alt empty (the hygiene review picks it up). Binary writes go through `s3.write_journal_image` (image-only allowlist, `journal/` sub-prefix; *not* an agent tool). `update-draft` runs `_gather_fills` (the blocking source pulls + image rehosting + the HTML-preview write) via `asyncio.to_thread`.
- **Browser-viewable HTML previews.** `update-draft` writes `draft.html` alongside `draft.md` via `tools.render` (Python-Markdown → a self-contained styled page; the CSS mirrors `content/buttondown/newsletter/buttondown-email.css`; block markers stripped + a "DRAFT…" banner). `draft.html` is the **only** review surface — `archive.md` / `buttondown.md` / `transcript/*.txt` are ship-shaped artifacts for their destinations and have no rendered HTML twin. `/eddy issue reorder` additionally writes `final-proposal.html` (the side-by-side editorial-review preview). Uploaded with `Cache-Control: no-cache` and a CloudFront invalidation of that path (`tools.cdn.invalidate`, distribution `WEEKLY_THING_CDN_DISTRIBUTION_ID`). Best-effort: a render/invalidation hiccup never fails the job. The pages carry a `<meta name="robots" content="noindex,nofollow">` and `files.thingelstad.com/robots.txt` is `Disallow: /`, so the in-progress drafts aren't indexed (they *are* publicly reachable by URL — same as `draft.md` always has been on that public prefix).
- **The draft.html editorial-review drawer.** When `update-draft` produces a review (`prompts/eddy/draft-review.md` — a solid pass: anchored, concrete *suggestions only*, never a rewrite — run via `_draft_review` on **Opus**, on every run with a real draft delta), `render.render_and_upload_html(..., review_md=…)` injects a fixed "Show review" button and a slide-in drawer on the right holding the rendered review. The drawer is **hidden by default** (`transform: translateX(100%)`, revealed by `body.rv-open` via a tiny inline `<script>` toggle) — so the same shareable URL reads as the clean draft until someone flips it on. When it's open and the viewport is wide enough (`min-width: 880px`), `body.rv-open` also shifts the draft column left (extra right margin reserved for the panel) so the review and the content sit side by side instead of the panel covering the text; on narrower windows the panel overlays. No review (empty draft → Eddy `PASS`, or no team) → no button, no drawer. The review prompt walks Intro → Notable → Journal → Briefly → Whole issue → **Hygiene** — that last pass is a deliverability-and-reader-skepticism lens (anchor text quality, anchor/domain mismatch, hype headings, spam-filter signal in voice, alt-text quality, subject-shaping section headings) folded into the same single Opus call, no separate scanner. This is now the **only** editorial review (the separate weekday `update-review.md` #editorial card was retired); the Build card surfaces its open-comment count + a link to this drawer.
- **Off-loop source pulls.** Jobs that hit external APIs from inside the bot's asyncio loop wrap the blocking calls in `asyncio.to_thread` so a slow Pinboard/Buttondown/Tinylytics/RSS/S3 round trip doesn't stall the gateway: `update-draft` (`_gather_fills` — Pinboard + micro.blog + image rehost + HTML render), `pinboard-scan` (`build_linky_context`), `daily-metrics` (campaign polls + `subscriber_growth` + `tinylytics.summary` + `build_marky_context`), `promotion-prep` (`buttondown.md` read + `build_marky_context`). Agent-loop calls (`bot.core`) are already thread-wrapped in `agent_loop.py`.
- **Marky operates on the last *published* issue**, not the in-flight one — the **Share-phase transition** (`put-to-bed`, which auto-fires `promotion-prep`) is the trigger; the issue is resolved via `db.get_latest_issue()` and Marky reads its `buttondown.md`.
- **Two archive search tools, not one.** `archive__search` is in-process BM25 (lexical — exact-word matches, cheap); `archive__retrieve` is semantic via Thingy's `/retrieve` (Bedrock Cohere embed + rerank, ~1s, ~$0.001/call). Prompts steer themes/concepts to `archive__retrieve`, specific phrases/person-names to `archive__search`. Six surfaces also pre-inject Thingy-grade context: pinboard-scan resonance, promotion-prep thread context, draft-review echoes, compose-subject thread awareness, linky research-card resonance, compose-echoes's mode-1 candidates. Shared helpers in `tools/archive_context.py` (fail-soft) + `tools/thingy_retrieve.py` (fail-loud — for compose-echoes's quality floor).
- **`update-draft` is a pure projection** — never an additive merge. Real authoring lives upstream. Re-running it gives the same output modulo upstream changes.
- **Job prompts get a `## Today` dynamic-context block.** Read it; don't recompute date math / counts / queue depth / goal progress.
