# workshop_bot вЂ” project memory

> Author-only Discord bot for *The Weekly Thing*. Helps Jamie assemble each week's issue and ships supporting bits (link curation, membership CTA, promotion drafts). One Python process, several `discord.py` clients (one per persona, own token/avatar), plus an APScheduler instance вЂ” all in the same asyncio loop.

Full design rationale: [`docs/workshop-content-loop-design-brief.md`](../../docs/workshop-content-loop-design-brief.md). The user-facing overview is [`README.md`](README.md). This file is the "what to keep in mind when working here" memory.

## Architecture: jobs are the spine

Every workshop_bot action вЂ” pulling content into the draft, reordering for the final, composing CTAs, drafting promotion copy вЂ” is a **job**: deterministic Python in `apps/workshop_bot/jobs/`, fired by the `/workshop вЂ¦` slash surface (host: Eddy) and/or by cron. Some jobs make small encapsulated LLM calls (Eddy's draft review, the compose-* jobs, Marky's reports); most are pure Python. The job is the unit of scheduling and on-demand execution.

The slash surface is organised around the *content artifact*, not around the word "job": `/workshop issue {start,update,status,final,haiku,subject,cta,publish}`, `/workshop links scan`, `/workshop promo {prep,metrics}`, `/workshop campaign {add,report,copy,sunset}`, `/workshop goal {set,done}`, plus a top-level `/workshop status`. The internal job names used below (and by the scheduler, the `job_locks` table, and `agent_runs`) are the dash-cased ones вЂ” `update-draft`, `create-final`, `compose-haiku`/`-meta`/`-cta`, `build-publish`, `pinboard-scan`, `promotion-prep`, `daily-metrics`, etc.; the slash names are the operator-facing aliases (`/workshop issue update` в†’ `update-draft`, `/workshop issue subject` в†’ `compose-meta`, `/workshop links scan` в†’ `pinboard-scan`, `/workshop promo metrics` в†’ `daily-metrics`, `/workshop goal done` в†’ `goal-achieved`, and so on).

`jobs/_base.py` is the runtime: `JobContext` (carries `deps` вЂ” corpus/registry/team вЂ” and a `trigger` label; has `channel()` / `post()` helpers), `JobResult` (`{ok, message, data}` вЂ” `message` is acked to the invoker), single-asset locking (`job_lock([assets], name)` context manager backed by the `job_locks` SQLite table вЂ” two jobs that write the same file can't overlap; a lock held by a dead pid is stolen since a restart gets a new pid), and the draft-block helpers (`replace_block` / `get_block` for the `<!-- block:NAME -->` вЂ¦ `<!-- /block:NAME -->` markers in `draft.md`).

Plus one top-level subcommand outside the artifact groups: **`/workshop status`** вЂ” a read-only DB-only ops snapshot (active issue window, active goal/campaigns, any held `job_locks`, the last few `agent_runs`). Distinct from `/workshop issue status` (the in-flight issue's *content* completeness) and `/workshop campaign report` (campaign perf vs expected). Source: `jobs/status.py`; `set-goal`/`goal-achieved`/`campaign-sunset`/`campaign-copy` are in `jobs/ops.py` (tiny, no-LLM).

Scheduler: `scheduler/jobs.py` declares the cron `JobSpec`s; `scheduler/handlers.py` has `content_job` (the bridge cronв†’jobs, wired as `functools.partial(content_job, job="<name>")`) and `rss_check` (poll the feed, dedupe via `agent_notes`, auto-fire `promotion-prep`). There are **no per-persona heartbeats** вЂ” everything an agent does on a cadence is a job.

Slash dispatch (`personas/commands.py`): the fast jobs `defer` в†’ run в†’ ack with the result (the followup send is wrapped in try/except вЂ” a Discord interaction token only lasts ~15 min). The **interactive** jobs (`create-final`, `compose-haiku`/`-meta`/`-cta` вЂ” they post options to a channel and wait for Jamie's reaction, which can outlast the token) instead ack *immediately* ("started вЂ” react in #editorial / #supporters") and then run; the job posts its own outcome to the channel, so no second followup is needed. `/workshop status` is attached directly to the `workshop` group rather than to one of the artifact subgroups (it's a bot-health view, not tied to a single artifact) вЂ” Discord allows a command group to hold both a subcommand and subcommand-groups.

### The unified asset pattern

Every piece of issue content is a standalone file in the S3 workspace `s3://files.thingelstad.com/weekly-thing/{N}/`:

- `intro.md` (the opener prose), `cover.md` (the cover photo's caption + `date  \nlocation` вЂ” `update-draft` prepends the derived `![](.../cover.jpg)`), `currently.md` (the optional "Currently" section) вЂ” Jamie writes (Drafts в†’ Shortcut); `update-draft` reads each verbatim into its block.
- `haiku.md` вЂ” `compose-haiku` writes (rendered bold + hard-break by `_base.format_haiku` at draft/publish time); `metadata.json` вЂ” `compose-meta` writes; `cta-1.md` / `cta-2.md` вЂ” `compose-cta` writes.
- `draft.md` вЂ” `update-draft` writes (a regenerable pure projection of all the above + upstream, shaped like a delivered issue).
- `final.md` вЂ” `create-final` writes (post-Eddy ordering; still block-structured).
- `publish.md` вЂ” `build-publish` writes (`---`-fenced parts: intro, cover block, the non-empty `## Header\n\n{content}` sections, CTAs, the haiku close; no block markers; the artifact `pipeline/content/content.py publish --issue N` pushes to Buttondown as a draft).
- `cover.jpg`, `cover-large.jpg`, `journal/*.jpg`, `body-{N}.mp3`, `weekly-thing-{N}.mp3` вЂ” written by the iOS Shortcuts and `pipeline/audio/`; agents can't touch them (the `tools/s3.py` extension allowlist is text-only). `eddy-edits.md` вЂ” Eddy, rarely, for a preserved revision proposal.

**Issue-markdown shape (mirrors the iOS-Shortcut bodies in `data/buttondown/bodies/`).** Top-level blocks are `---`-fenced, in order: intro prose в†’ cover (`![](.../cover.jpg)` + caption + `Month D, YYYY  \nLocation`) в†’ `## Currently` (rare; dropped when empty) в†’ `## Notable` в†’ `## Journal` в†’ `## Briefly` в†’ `A haiku to leave you withвЂ¦` + the haiku + the closing "discuss on Reddit" line + `рџ‘ЁвЂЌрџ’»`. The list loops have strict formatting:
- **Notable** вЂ” leads with `_You can discuss any of these links at the [Weekly Thing {N} tag in r/WeeklyThing](вЂ¦flair_name%3A%22Weekly%20Thing%20{N}%22)._` (carries the issue number вЂ” generated, not templated), one blank line, then items. Each item is `### [Title](url)` then `\n\n{commentary}`. **Two blank lines between items.**
- **Briefly** вЂ” each item is `{commentary} в†’ **[Title](url)**` (commentary first, then the `в†’` arrow, then the bolded link; no headings). **One blank line between items.**
- **Journal** вЂ” a status update (no title) is `[Mon D, YYYY at H:MM AM/PM](url)\n\n{content}`; a *titled* post is "elevated": `### [Title](url)  \n{Mon D, YYYY at H:MM AM/PM}\n\n{content}` (H3 link with a markdown hard break, date plain on the next line). Photos already live inside `content_md` (rehosted), each on its own paragraph. **Two blank lines between entries.**
- **Haiku** вЂ” `**line one  \nline two  \nline three**` (bold-wrapped, markdown hard break between lines).

Renderers live in `jobs/update_draft.py` (`_render_notable` / `_render_brief` / `_render_journal`, `_reddit_tag_line`); `_base.format_haiku` is shared with `build-publish`. The membership CTA and the email tracking pixel are *not* in `draft.md` (the CTA is composed separately and assembled into `publish.md`; the pixel is a Buttondown artifact). `update-draft` rebuilds `draft.md` from `templates/draft_starter.md` every run, so the layout always matches the template.

`tools/draft.py` (`section_status`) parses the draft for section item counts + asset presence + what's still missing for ship; `draft__section_status` is the agent tool over it. `tools/context.py` builds the per-persona dynamic-context blocks (`build_eddy_context` / `build_linky_context` / `build_patty_context` / `build_marky_context`) вЂ” runtime-computed facts injected into job prompts so the model doesn't recompute date math, word counts, queue depths, goal progress.

`tools/interaction.py` is the reaction primitive (`await_choice` / `await_approval`) вЂ” post options, wait for Jamie's reaction (filtered on `DISCORD_OWNER_USER_ID`), return the pick. Jobs hold their asset lock for the duration of the interaction.

## Storage вЂ” two layers, hard boundary

- **Public S3** `s3://files.thingelstad.com/weekly-thing/{N}/` вЂ” only what ships (text/JSON via the `tools/s3.py` allowlist, plus binaries written by other pipelines). Backs the `workspace__*` tools (`workspace__list_all` / `list_files` / `read` / `write`).
- **SQLite** `apps/workshop_bot/data/workshop.db` вЂ” everything else. Tables: `agent_outputs`, `agent_runs`, `agent_notes`, `issue_windows`, `link_candidates`, `pinboard_popular_seen`, `pinboard_research_done`, `subscriber_events_seen`, `thingy_tokens`, `thingy_requests`; **new in the content-loop redesign:** `job_locks`, `draft_digests`, `goals`, `campaigns`, `campaign_metrics`.

There is **no private workshop bucket** anymore. Decommissioned in the redesign: `s3://weekly-thing-workshop/`, the `WORKSHOP_BUCKET` env var, `tools/persona_s3.py` + the `s3_personas__*` tools, `tools/inbox.py` + the `agent_inbox` table + the `inbox__*` tools, the reserved-but-never-wired `analytics` / `supporter_events` / `channel_routes` tables, the `/workshop heartbeat` and `/workshop next-issue` slash commands. `s3_issues__*` was renamed to `workspace__*`. `db/schema.sql` carries a `DROP TABLE IF EXISTS` for each dropped table so a long-lived DB converges with a fresh install.

## Conventions worth remembering

- **Pinboard tag convention is strict: only `_brief`.** Among the bookmarks `update-draft` projects into the issue: untagged-in-window = Notable; `_brief` = Briefly. The old `_featured` tag/section is retired. `pinboard__capture_blurb` writes a blurb verbatim + tags `_brief` + clears `toread` atomically. **`client.issue_window_candidates` only returns *public, read* bookmarks** вЂ” anything still flagged `toread` (Jamie hasn't worked through it) or marked private (`shared=no`) is excluded, so the draft never picks up an unreviewed link. (Linky's `pinboard-scan` still *sees* the `toread` pile вЂ” that's its job; the exclusion is only in the draft projection.)
- **`pinboard__popular_unseen` filters avoid-domains.** Before the `pinboard_popular_seen` dedup it drops items whose host is on `tools/avoid_domains.py`'s exclusion set (own domains, Buttondown, image CDNs, URL shorteners, social, Wikipedia, YouTube/Spotify/Vimeo, generic CDNs) вЂ” so Linky never even considers a "popular on Pinboard" Wikipedia article or t.co redirect as a possible Notable. That list is a hand-maintained *copy* of `pipeline/content/domain_exclusions.py` (`pipeline/` isn't importable from here); keep them loosely in sync вЂ” an exact match isn't required since this only gates Linky's scan, nothing it touches ships.
- **micro.blog: pull everything in window, no tag filtering** (`tools/microblog.py`). Uses the Micropub source query (`GET {MICROBLOG_MICROPUB_URL}?q=source`, `Authorization: Bearer MICROBLOG_API_KEY`) вЂ” returns posts as mf2-JSON with `properties.content` carrying the **native markdown Jamie wrote** (a string; micro.blog embeds photo uploads as `<img src=вЂ¦>` tags inside it). `MICROBLOG_API_KEY` is **required** вЂ” no fallback; if micro.blog is down, `journal.fill` leaves a placeholder line (Eddy's review flags it). Filtering/curation happens at `create-final` (Eddy can cut Journal entries).
- **Journal images are rehosted** (`tools/journal_images.py`, called from `journal.fill`). Each `<img>` / `![]()` on an upload host (`MICROBLOG_IMAGE_HOSTS`, default `www.thingelstad.com` `/uploads/`, `micro.thingelstad.com`, `cdn.uploads.micro.blog`, `uploads.micro.blog`) is downloaded, resized to `MICROBLOG_IMAGE_MAX_DIM` (default 600px long side, downscale-only; JPEG q80 / PNG optimized), and copied into `weekly-thing/{N}/journal/<name>` (reusing the micro.blog hash basename), then the reference is rewritten to the local URL and `<img>` tags are normalized to `![alt](url)`, each on its own paragraph (so adjacent gallery `<img><img>` don't run together). A HEAD-skip makes daily re-runs cheap; a per-image failure leaves that one's original URL alone. Binary writes go through `s3.write_journal_image` (image-only allowlist, `journal/` sub-prefix; *not* an agent tool). `update-draft` runs `_gather_fills` (the blocking source pulls + image rehosting + the HTML-preview write) via `asyncio.to_thread`.
- **Browser-viewable HTML previews.** `update-draft` / `create-final` / `build-publish` write a `.html` twin alongside their `.md` (`draft.html` / `final.html` / `publish.html` in the workspace) via `tools.render` (Python-Markdown в†’ a self-contained styled page; the CSS mirrors `content/buttondown/newsletter/buttondown-email.css`; `draft`/`final` strip the block markers and get a "DRAFTвЂ¦/FINALвЂ¦" banner, `publish` renders clean). They're uploaded with `Cache-Control: no-cache` and a CloudFront invalidation of that path (`tools.cdn.invalidate`, distribution `WEEKLY_THING_CDN_DISTRIBUTION_ID`, default the prod one; set to empty to disable) вЂ” so Jamie always sees the latest at `https://files.thingelstad.com/weekly-thing/{N}/draft.html` etc. The URL is surfaced in the job's result and (for `update-draft`, TueвЂ“Fri) appended to Eddy's review card. Best-effort: a render/invalidation hiccup never fails the job. The pages carry a `<meta name="robots" content="noindex,nofollow">` and `files.thingelstad.com/robots.txt` is already `Disallow: /`, so the in-progress drafts aren't indexed (they *are* publicly reachable by URL вЂ” same as `draft.md` always has been on that public prefix).
- **The draft.html editorial-review drawer.** When `update-draft` produces a review (`prompts/eddy/draft-review.md` вЂ” a solid pass: anchored, concrete *suggestions only*, never a rewrite вЂ” run via `_draft_review` on sonnet, every run), `render.render_and_upload_html(..., review_md=вЂ¦)` injects a fixed "Show review" button and a slide-in drawer on the right holding the rendered review. The drawer is **hidden by default** (`transform: translateX(100%)`, revealed by `body.rv-open` via a tiny inline `<script>` toggle) вЂ” so the same shareable URL reads as the clean draft until someone flips it on. No review (empty draft в†’ Eddy `PASS`, or no team) в†’ no button, no drawer. The Discord `#editorial` card (`update-review.md`, TueвЂ“Fri) is separate and unchanged вЂ” different audience, different cadence.
- **Off-loop source pulls.** Jobs that hit external APIs from inside the bot's asyncio loop wrap the blocking calls in `asyncio.to_thread` so a slow Pinboard/Buttondown/Tinylytics/RSS/S3 round trip doesn't stall the gateway: `update-draft` (`_gather_fills` вЂ” Pinboard + micro.blog + image rehost + HTML render), `pinboard-scan` (`build_linky_context`), `daily-metrics` (campaign polls + `subscriber_growth` + `tinylytics.summary` + `build_marky_context`), `promotion-prep` (RSS lookup + `publish.md` read + `build_marky_context`). Agent-loop calls (`bot.core`) are already thread-wrapped in `agent_loop.py`.
- **Marky operates on the last *published* issue**, not the in-flight one вЂ” the RSS feed (`tools/rss.py`) is her trigger; she reads `publish.md`.
- **`update-draft` is a pure projection** вЂ” never an additive merge. Real authoring lives upstream. Re-running it gives the same output modulo upstream changes.
- **Job prompts get a `## Today` dynamic-context block.** Read it; don't recompute date math / counts / queue depth / goal progress.
- Prompt name в†’ file: `team` в†’ `prompts/shared/team.md`; `<persona>` в†’ `prompts/<persona>/prompt.md`; `<persona>-<file>` в†’ `prompts/<persona>/<file>.md` (e.g. `eddy-update-review`, `linky-pinboard-scan`, `marky-promotion-prep`, `patty-compose-cta`). Edits need a bot restart (prompts are cached at first read).
- New schema в†’ `db/schema.sql` (idempotent вЂ” re-runs every boot; the `goals` seed uses `WHERE NOT EXISTS`); new *columns* on existing tables в†’ `db._COLUMN_MIGRATIONS`.
- Tests: `python -m unittest discover -s apps/workshop_bot/tests -t .` вЂ” discord/anthropic/httpx stubbed (`tests/_stubs.py`); S3/Pinboard/micro.blog/Anthropic stubbed per-test. ~300 tests. Don't hit the network in tests.
